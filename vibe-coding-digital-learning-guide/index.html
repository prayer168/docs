一、為什麼一位自然科教師會開始 Vibe Coding？

最初的願望通常很單純：我想讓學生「看見」課本裡不容易看見的現象。

教光的折射時，課本會畫一條光線穿過水面，學生會背「斜斜射入會偏折」，卻不一定真正理解入射角、折射角、法線與介質的關係。我想做一個網頁，讓學生拖曳雷射筆、切換空氣、水和玻璃，親手改變入射角，再從畫面與數值找出規律。問題是，我懂教學，不一定熟悉 HTML、CSS 和 JavaScript；知道學生需要什麼，卻不知道如何把腦中的畫面變成程式。

現成教材往往功能完整，卻不一定符合班級語言、課程進度或操作裝置。改別人的程式又常牽一髮動全身：電腦上看似漂亮，手機卻出現水平捲軸；滑鼠可以拖曳，平板手指卻毫無反應；改了一句回饋，重設按鈕突然失效。即使教材做完，還有檔案路徑、GitHub Pages、快取與日後維護等關卡。

Vibe Coding 是什麼？
它是一種以自然語言描述意圖，讓 AI 協助產生、解釋、修改與測試程式的開發方式。它降低了開始的門檻，但沒有取消思考、驗證與負責。教師真正的優勢，是能定義學習目標、判斷科學內容、觀察學生卡在哪裡，再把問題說清楚。

第一次生成不完美很正常。Vibe Coding 的核心不是「一次神提示詞」，而是一個循環：描述需求、做出小版本、實際操作、留下證據、精準回報、局部修正、重新測試。當這個循環穩定，AI 才從會吐程式碼的工具，變成可合作的開發夥伴。

二、寫程式以前，先完成教材需求訪談

最常見的失敗不是 AI 不夠聰明，而是需求只有一句「幫我做光的折射教材」。AI 不知道對象是三年級還是國中、課堂只有十分鐘還是一節課，也不知道要精確計算史涅爾定律，或只做定性觀察。資訊空白處越多，AI 自由猜測越多。

三、第一版提示詞：不是越長越好，而是邊界越清楚越好

有效提示詞包含十二個部分：角色、教材目標、對象、內容、功能、操作流程、視覺、技術規格、檔案結構、驗收、錯誤處理與部署。它們不是華麗格式，而是把「不准 AI 猜的事情」先決定。

模糊版

「幫我做一個光的折射互動教材，要漂亮、可以拖曳。」

AI 可能自行決定年級、介質、公式、畫面比例與拖曳方式；完成後看似有動效，卻可能把角度相對水面計算，或者只支援滑鼠。

可驗收版

指定國小高年級、角度以法線為基準、介質與折射率、Pointer Events、360px RWD、重設行為、不得使用 CDN，並列出五個測試案例。

結果不一定一次完美，但錯誤範圍縮小，也能判定「完成」而不是只憑感覺。

寫提示詞的重要順序：先寫學習目標與科學規則，再寫核心操作與驗收；視覺風格排在後面。教材不是先漂亮，再設法塞入教學。
範本 A｜單一 HTML 小型互動教材
你是熟悉國小自然科、前端開發與無障礙設計的教材工程師。請製作「光的折射實驗室」，供國小高年級兩人一組平板操作 20 分鐘。學生可調整入射角並切換空氣、水、玻璃；角度一律相對法線，依 n1 sinθ1 = n2 sinθ2 計算。需標示入射光、折射光、法線、介面、入射角與折射角，並用國小可理解的文字回饋。以 Pointer Events 同時支援滑鼠與觸控，360px 起不得整頁水平溢出，按鈕至少 44px。請先完成 MVP：一項核心互動、即時數值、觀察提問與可靠重設。所有 HTML、CSS、JavaScript 放入單一 index.html，不用 CDN。完成後依五組角度測試、重設、觸控與 RWD 自我檢查，再輸出完整檔案。
範本 B｜HTML、CSS、JavaScript 分離專案
請建立可維護的光折射教材專案，檔案為 index.html、styles.css、app.js、README.md、TESTS.md。先列出資料模型、狀態、函式責任和檔案結構，再實作。介質資料集中管理，不把折射率散落在事件處理函式。輸入需驗證；重設函式要還原角度、介質、紀錄、回饋和動畫。請提供桌機、平板、手機及 Chrome、Edge、Safari 的測試清單。不要加入登入、音效或資料庫。
範本 C｜Supabase 學習紀錄進階版
在已驗證的光折射 MVP 上新增 Supabase 學習紀錄。開始前先閱讀既有檔案，不重寫模擬核心。請設計最少資料欄位，只保存課程代碼、匿名組別代碼、介質、入射角、折射角、預測、時間戳，不收集姓名。提供 SQL schema、RLS 政策與前端錯誤處理。前端只能使用可公開的 anon key，不得放 service_role 或其他秘密。寫入失敗時教材仍可操作，顯示可理解訊息並可匯出本機紀錄。列出安全、離線、重複送出與權限測試。

MVP（Minimum Viable Product，最小可行產品）在教材裡可譯為「最小可行教材」：基本畫面、一項核心互動、正確科學內容、基本回饋和可重設。它不是粗糙成品，而是最早可以驗證教學假設的版本。

如果一開始同時加入動畫、音效、闖關、計時、登入、後台與資料庫，當數值錯誤時，很難知道是公式、畫面座標、狀態同步，還是動畫延遲造成。功能越多，錯誤組合不是相加，而是相乘。先確認「改變入射角後，折射光方向與數值正確」，再加觀察紀錄；接著才考慮遊戲化與後端。

MVP 必須有

  • 光線與法線清楚
  • 角度或拖曳可以控制
  • 介質切換正確
  • 數值與圖形一致
  • 重設回到同一初始狀態

第二階段再加

  • 動畫、音效、闖關與計時
  • 學習單、紀錄、匯出
  • 教師模式與班級資料
  • 登入、資料庫與分享

五、貫穿案例：光的折射互動教材

第一次生成：看起來會動，不等於科學正確

初版畫面很吸睛：雷射光穿過藍色水面，拖曳也很流暢。但檢查後發現角度是相對水面而非法線；從空氣進入水時，有些角度反而離法線更遠；玻璃折射率雖標成 1.50，圖形卻使用固定偏移量。這是教材開發最危險的錯誤——畫面越漂亮，學生越容易相信。

科學校正：史涅爾定律為 n₁sinθ₁=n₂sinθ₂。角度 θ 是光線與「介面法線」的夾角。從低折射率進入高折射率，折射光通常向法線偏；由高到低則離法線偏，且可能發生全反射。折射率會隨波長、溫度與材料而異，教材中的 1.00、1.33、約 1.50 應標為教學近似值。

第二個問題：滑鼠能拖，平板不能拖

原因是初版只監聽 mousedown/mousemove。改用 Pointer Events（指標事件）統一滑鼠、手指與觸控筆，並在拖曳區使用適當的 touch-action。修正時還要保留角度滑桿作為替代操作，讓鍵盤與精細動作不便的學生仍能使用。

第三個問題：修了手機版，重設卻失效

AI 為了重排畫布，改了元素 id，原本的重設函式仍尋找舊 id。這提醒我們：修改畫面也可能破壞 JavaScript。處理方式不是「全部重寫」，而是先用 Console 取得錯誤、比對事件綁定與 DOM id，做最小修正，然後測試拖曳、介質切換、紀錄、重設與三種尺寸。

版本與課堂回饋

穩定版保存為 v0.1-mvp；修正角度與全反射為 v0.2-science；完成觸控與 RWD 為 v0.3-mobile；課堂試用後，學生反映法線不明顯、題目不知道先做什麼，於是 v0.4 加粗法線、加入「先預測、再操作、最後解釋」三步提示。這就是教材從程式作品走向學習工具的時刻：版本更新的理由來自學生的學習證據，而不是只追求更多功能。

七、向 AI 回報錯誤：把「怪怪的」變成可重現的證據

好的除錯回報至少包含：預期結果、實際結果、重現步驟、裝置與瀏覽器、Console 訊息、截圖或錄影、相關程式碼、問題前的修改，以及禁止破壞的功能。Console 是瀏覽器提供的錯誤與執行訊息視窗；Chrome／Edge 可按 F12 開啟開發者工具。

標準除錯提示詞
請先分析問題原因，不要立即重寫整個專案。找出最可能出錯的檔案、函式與程式碼範圍，說明修改方式，再進行最小幅度修正。預期結果：＿＿。實際結果：＿＿。重現步驟：1.＿＿ 2.＿＿ 3.＿＿。裝置／瀏覽器：＿＿。Console 訊息：＿＿。發生前修改：＿＿。必須保留：＿＿。修改後請檢查是否影響其他功能，並列出可由我操作的測試步驟。

八、教材修改：先判斷規模，再決定策略

規模例子策略
小修改文字、顏色、按鈕名稱、間距、單一數值指定元素與目標值，局部改動；快速視覺與功能測試
中型修改新操作步驟、提示回饋、題型、重設、手機版先列影響範圍與狀態變化；修改後做回歸測試
大型修改重做流程、拆檔、登入、資料庫、教師後台、換框架保留穩定版，建立分支或副本；訂遷移計畫與分階段驗收

修改前七件事：備份可用版、記錄版號、一次處理一個主問題、寫出不可變動項目、修改後回歸測試、跨裝置確認、保留回復路徑。若 AI 建議重寫整個檔案，要先問「為什麼局部修正不足？哪些行為可能改變？」

最安全的節奏：一個修改目的 → 一個可比較版本 → 一份測試結果 → 再進下一項。不要把十張便利貼一次倒進提示詞。

九、避免 AI 越修越亂：讓專案擁有自己的記憶

長對話會遺漏早期條件；貼到 AI 的可能是舊檔；不同工具輪流改寫，命名與架構逐漸分裂。解法不是期待 AI 永遠記得，而是把專案真相寫進檔案。

requirements.md

教材目標、功能、科學規則、裝置、不可變動項目與驗收標準。

README.md

如何開啟、檔案結構、主要函式、部署與維護方式。

CHANGELOG.md

每版新增、修改、修正與已知限制；日期與版號不可少。

BUGS.md／TESTS.md

可重現問題、證據、狀態、修正結果與每次必跑的測試案例。

使用 Git 保存版本：每完成可操作的小步驟就提交；穩定階段標上 v0.1、v0.2。大型重構放在獨立分支。每次交給 AI 前，先要求它閱讀最新需求與檔案、摘要理解、列影響範圍，再編輯。若換到新對話，用交接提示詞附上版本、完成項目、已知問題、禁止變動與下一個唯一任務。

十、測試與驗收：修好新功能，不代表教材仍然完整

回歸測試是指修改後重新檢查原本正常的功能。光折射教材修正手機版後，不能只看手機畫面；仍要測試折射計算、介質切換、全反射判斷、紀錄與重設。最省時間的方式，是建立固定測試清單。

三層驗收：程式層確認沒有錯誤；科學層確認公式、單位、圖形與文字一致；教學層確認學生知道要做什麼、能解釋觀察，而不只是把滑桿拉來拉去。

十一、從完成到上線：部署不是最後按一下按鈕

  1. 整理檔案，首頁必須為小寫 index.html。
  2. 檢查圖片、CSS、JavaScript 的相對路徑與大小寫。
  3. 移除測試資料、個資、service role key 與其他秘密。
  4. 上傳 GitHub，選定分支與資料夾，開啟 Pages。
  5. 等待部署完成，以正式網址而非本機檔案測試。
  6. 用無痕視窗排除快取，再用手機與平板實測。
  7. 記錄網址、版號、日期、已知限制與回復方式。

十三、教師可重複使用的 Vibe Coding SOP

  1. 定義教學問題：學生現在不懂什麼？互動如何產生證據？
  2. 填需求表：對象、情境、核心操作、科學依據與驗收。
  3. 寫第一版提示詞：把不可猜測的邊界說明。
  4. 生成 MVP：只做一項核心互動與重設。
  5. 教師校正：先查科學內容，再看美觀。
  6. 小規模試用：觀察學生語言、手勢與卡點。
  7. 建立問題清單：現象、重現步驟、影響程度與證據。
  8. 一次修一類：最小修改，不順手大改。
  9. 回歸測試：新功能與舊功能一起測。
  10. 保存版本：Git 提交、版號、修改紀錄。
  11. 部署上線：正式網址跨裝置檢查。
  12. 蒐集回饋：問學生如何判斷，不只問喜不喜歡。
  13. 更新紀錄：每次改版都回答「為何改、改了什麼、如何驗證」。

走到這裡，教師不是被 AI 取代的程式設計師，而是教學設計者、需求定義者、科學內容審查者、學習體驗測試者、問題判斷者與品質把關者。真正的進步，不是學會叫 AI 多寫幾百行，而是能把教材變成一個有需求、有證據、有版本、有責任邊界的專案。

最後的核心能力：說清楚、拆得小、看得懂證據、守住科學正確、留下可回復版本，並以學生是否真正學會作為最終驗收。

常見問答

我完全不會程式，可以開始嗎？

可以從單一 HTML、單一互動開始，但要學會開啟 Console、辨識檔案與備份版本。AI 可以解釋程式，最終仍需你判斷教材是否正確。

每次都請 AI 輸出完整程式比較安全嗎？

初版可以；進入維護後，完整重生常把穩定功能一起改掉。優先局部修正，重大重構才建立新分支並完整比較。

Supabase anon key 可以放前端嗎？

anon key 的設計用途可以是前端公開識別，但安全性必須由 RLS 與資料庫政策保護；service_role 等秘密絕不能放入前端。

為什麼本機雙擊正常，GitHub Pages 卻壞掉？

最常見是檔名大小寫、相對路徑、子目錄基底、缺少檔案、HTTP 資源被 HTTPS 阻擋或快取。先看正式網址的 Console 與 Network。
