入門基礎 1基礎開發與版本控制

1. Repository / Repo｜儲存庫

專案的完整程式碼與文件空間。

工作原理：保存原始碼、設定、測試、文件與歷史版本。

應用場景：AI Agent 讀取專案結構、定位入口檔、決定修改範圍。

2. Commit｜提交

一次可追蹤的修改紀錄。

工作原理：Git 將檔案差異、作者、時間與說明記成版本快照。

應用場景：讓 AI 修改有還原點，也方便寫 changelog。

3. Branch｜分支

從主線分出的開發路線。

工作原理：不同分支可以平行修改，完成後再合併。

應用場景：讓 AI 在 feature/fix 分支工作，避免破壞 main。

4. Pull Request / PR｜合併請求

把分支修改送出審查。

工作原理：PR 集中呈現 diff、討論、檢查結果與合併紀錄。

應用場景：AI 完成修改後開 PR，交給人類或另一個 AI review。

5. Diff｜差異比較

修改前後的行級差異。

工作原理：顯示新增、刪除、異動的程式碼。

應用場景：檢查 AI 到底改了什麼，避免被摘要誤導。

入門基礎 2Prompt 與上下文工程

6. Prompt｜提示詞

給 AI 的任務說明。

工作原理：模型根據 prompt、上下文與工具結果產生回應。

應用場景：描述需求、限制、輸出格式、測試方式。

7. System Prompt｜系統提示詞

高優先級的行為規則。

工作原理：設定角色、規範、語氣、邊界與工具使用策略。

應用場景：規定 AI 只能做最小修改、必須列出測試。

8. Context｜上下文

AI 當下可見的所有資訊。

工作原理：包含對話、程式碼、文件、錯誤訊息與工具回傳。

應用場景：提供完整上下文可大幅降低亂猜。

9. Context Window｜上下文視窗

模型一次能處理的文字容量。

工作原理：超過容量時需要摘要、切片或檢索。

應用場景：大型專案要分階段讀檔，不能全部塞進去。

10. Few-shot Prompting｜少樣本提示

提供範例讓 AI 模仿。

工作原理：模型從例子推斷格式與風格。

應用場景：固定 commit message、題庫 JSON、教案格式。

11. Context Engineering｜上下文工程

設計 AI 看見什麼、何時看、看多少。

工作原理：比單純 prompt 更重視資料選擇、壓縮、工具回傳與記憶。

應用場景：大型 Agent 要管理任務狀態、檔案摘要與檢索結果。

12. Prompt Template｜提示詞模板

可重複套用的 prompt 結構。

工作原理：保留角色、任務、限制、輸出格式，填入變數。

應用場景：建立「除錯模板」「重構模板」「文章模板」。

13. Instruction Hierarchy｜指令階層

不同來源指令的優先順序。

工作原理：系統、開發者、使用者、工具資料的權重不同。

應用場景：防止外部 README 覆蓋使用者任務。

14. Role Prompting｜角色提示

指定 AI 扮演某種專家。

工作原理：角色會影響檢查角度與輸出風格。

應用場景：資深前端、資安審查員、自然科教材設計師。

15. Constraint｜限制條件

要求 AI 必須遵守的邊界。

工作原理：例如不可改資料庫、不可新增套件、不可重寫全文。

應用場景：AI Coding 最重要的安全護欄之一。

16. Output Format｜輸出格式

要求回答的固定格式。

工作原理：可指定表格、JSON、步驟、檔案清單。

應用場景：讓 AI 回報修改內容時更容易檢查。

17. Self-critique｜自我檢查

要求 AI 先檢查自己的方案。

工作原理：模型產出後再從風險、測試、遺漏角度檢討。

應用場景：提交前請 AI 列出可能破壞點。

18. Plan-then-Act｜先規劃後執行

先產生計畫，再修改檔案。

工作原理：降低 Agent 無目標亂改的風險。

應用場景：大型任務要求 AI 先列分步計畫。

19. Guardrail Prompt｜護欄提示

限制 AI 行為的提示規則。

工作原理：明定禁止事項、審核點與失敗時處理。

應用場景：禁止刪除檔案、禁止改 package lock。

入門基礎 3LLM 與生成模型

20. LLM｜大型語言模型

能理解與生成文字、程式碼的模型。

工作原理：根據上下文預測高機率輸出。

應用場景：解釋錯誤、寫函式、補文件、重構程式。

21. Token｜詞元

模型處理文字的基本單位。

工作原理：文字會被切成 token，用於計算上下文與成本。

應用場景：估算長文件是否需要拆分。

22. Temperature｜溫度

控制輸出隨機程度。

工作原理：低溫較穩定，高溫較有創意但風險高。

應用場景：寫程式低溫；命名、文案可稍高。

23. Hallucination｜幻覺

AI 產生看似合理但錯誤的內容。

工作原理：模型不是資料庫，上下文不足時可能編造。

應用場景：要求 AI 列出實際檔案、測試與來源。

入門基礎 4Agent 與自主工作流

24. Agent｜代理人

能規劃、使用工具並執行任務的 AI。

工作原理：結合模型、工具、記憶、環境與回饋迴圈。

應用場景：自動讀 repo、改檔、跑測試、開 PR。

25. Coding Agent｜程式代理

專門處理軟體開發任務的 Agent。

工作原理：可讀寫檔案、執行 shell、檢查測試結果。

應用場景：Codex、Claude Code、Copilot agent、Cursor agent。

26. Vibe Coding｜氛圍式寫程式

用自然語言驅動 AI 快速開發。

工作原理：人類決定方向與審查，AI 產生原型與迭代。

應用場景：教材網站、HTML 工具、小型 app 原型。

27. Orchestrator｜編排者

負責拆任務與分派工作的主控角色。

工作原理：把大任務拆成設計、實作、測試、文件。

應用場景：多模型協作或多 Agent 分工。

28. Human-in-the-loop｜人類在迴圈中

關鍵決策需要人類確認。

工作原理：AI 可建議與執行低風險步驟，高風險步驟需審核。

應用場景：部署、刪除資料庫、寄信、金鑰操作前確認。

29. Agent Loop｜代理迴圈

AI 反覆觀察、思考、行動、檢查。

工作原理：Observe → Plan → Act → Reflect → Repeat。

應用場景：長任務修 bug、跑測試、根據結果再修。

30. ReAct｜Reason + Act 模式

推理與行動交替進行的 Agent 方法。

工作原理：先判斷下一步，再呼叫工具，根據結果再決策。

應用場景：搜尋資料、讀檔、執行測試時常見。

31. Reflection｜反思步驟

AI 對產出進行二次檢查。

工作原理：檢查是否符合需求、是否漏測、是否有風險。

應用場景：修改完成後要求 AI 自評。

32. Task Decomposition｜任務拆解

把大任務拆成小步驟。

工作原理：每一步有輸入、輸出、檢查點。

應用場景：把「做網站」拆成架構、內容、樣式、互動、測試。

33. Sub-agent｜子代理

專門負責某類任務的 Agent。

工作原理：主代理分派給測試、文件、資安、前端子代理。

應用場景：多模型協作工作流。

34. Multi-agent｜多代理系統

多個 Agent 合作完成任務。

工作原理：不同角色互相審查、補位、分工。

應用場景：一個寫程式、一個審查、一個寫文件。

35. Memory｜記憶

AI 保存的長期或短期資訊。

工作原理：可保存偏好、專案規則、決策、進度。

應用場景：記住你的 HTML 技術文件風格與命名習慣。

36. Scratchpad｜暫存工作區

AI 用來整理中間資訊的空間。

工作原理：保存待辦、假設、檔案摘要、測試結果。

應用場景：長任務避免忘記前面已檢查的內容。

37. Checkpoint｜檢查點

任務中的可回復狀態。

工作原理：在重要修改前後保存狀態。

應用場景：大型 refactor 前先建立 git commit。

38. Autonomy Level｜自主程度

AI 可自行決策與執行的範圍。

工作原理：從只建議，到可改檔，到可部署，風險逐步升高。

應用場景：依任務風險設定審核門檻。

入門基礎 5工具呼叫與結構化輸出

39. Function Calling｜函式呼叫

模型選擇並填寫函式參數。

工作原理：LLM 產生結構化呼叫，由外部程式執行。

應用場景：查日曆、建立檔案、呼叫 API。

40. Tool Calling｜工具呼叫

AI 使用外部工具完成任務。

工作原理：工具可包含搜尋、檔案、終端機、瀏覽器、資料庫。

應用場景：AI 跑 npm test、讀 PDF、查 GitHub issue。

41. Structured Output｜結構化輸出

要求 AI 產生固定格式。

工作原理：使用 JSON、表格、schema 降低解析錯誤。

應用場景：自動生成題庫、行事曆、設定檔。

42. JSON Schema｜JSON 結構規格

描述 JSON 欄位與型別的規則。

工作原理：可驗證必要欄位、格式、巢狀結構。

應用場景：讓 AI 產生可被程式穩定讀取的資料。

入門基礎 6MCP 與外部系統

43. MCP｜Model Context Protocol

AI 應用連接外部資料與工具的開放標準。

工作原理：像 AI 的 USB-C，讓模型存取檔案、資料庫、工具與工作流。

應用場景：把 GitHub、Drive、資料庫、校本教材庫接到 AI。

44. MCP Server｜MCP 伺服器

提供工具與資源給 AI 的服務。

工作原理：宣告可用工具、資源與提示，讓 AI client 呼叫。

應用場景：建立自然科教材查詢 server。

入門基礎 7RAG 與知識庫

45. RAG｜檢索增強生成

回答前先檢索資料再生成。

工作原理：查詢相關文件片段，交給 LLM 根據來源回答。

應用場景：打造教案、研習、科展資料知識庫。

46. Embedding｜嵌入向量

把文字轉成語意數字向量。

工作原理：語意相近的文字向量距離較近。

應用場景：搜尋「馬達不轉」也能找到「無法運作」。

47. Vector Store｜向量資料庫

儲存與搜尋 embedding 的資料庫。

工作原理：以相似度找出最相關片段。

應用場景：建立黑熊老師自然科學知識庫。

48. Chunk｜文件切片

把長文件切成小段。

工作原理：讓檢索能取回精準段落，而不是整份文件。

應用場景：PDF、教案、技術文件分段索引。

49. Retrieval｜檢索

根據問題找出相關資料。

工作原理：可用關鍵字、向量搜尋、混合搜尋與 metadata。

應用場景：查比賽、法規、教材段落。

50. Grounding｜依據來源回答

讓 AI 回答有可驗證根據。

工作原理：把輸出限制在來源內容與檢索結果上。

應用場景：技術文章、政策建議、期刊投稿需要引用。

入門進階 1測試、除錯與品質

51. Bug｜錯誤

程式沒有照預期運作。

工作原理：可能源自語法、邏輯、環境、資料或依賴。

應用場景：HTML 打不開、簡報亂碼、App 閃退。

52. Debug｜除錯

找出錯誤原因並修正。

工作原理：重現問題、讀錯誤、縮小範圍、修正、驗證。

應用場景：要求 AI 先列假設再做最小修改。

53. Stack Trace｜錯誤堆疊

錯誤發生時的呼叫路徑。

工作原理：顯示檔案、行號、函式與例外訊息。

應用場景：貼給 AI 可快速定位問題。

54. Unit Test｜單元測試

測試單一函式或模組。

工作原理：用小輸入驗證輸出是否正確。

應用場景：測日期轉換、表單驗證、資料解析。

55. Integration Test｜整合測試

測試多模組串接是否正常。

工作原理：檢查前端、後端、API、資料庫互動。

應用場景：表單送出後寫入 Supabase 並顯示成功。

56. Regression Test｜回歸測試

確認新修改沒有弄壞舊功能。

工作原理：每次修改後重跑既有測試。

應用場景：AI 改首頁後確認搜尋、複製、RWD 沒壞。

57. Eval｜模型評估

用固定任務評估 AI 表現。

工作原理：衡量正確性、格式、引用、安全與可執行性。

應用場景：建立 AI 產生教案或程式碼的評分集。

58. Lint｜靜態程式檢查

檢查語法、風格與潛在錯誤。

工作原理：不執行程式也能檢查程式碼問題。

應用場景：ESLint、Ruff、flake8。

59. Formatter｜格式化工具

自動整理程式排版。

工作原理：統一縮排、空格、換行與引號風格。

應用場景：Prettier、Black。

60. Type Checking｜型別檢查

檢查資料型別是否一致。

工作原理：在執行前發現函式參數、回傳型別錯誤。

應用場景：TypeScript、mypy、pyright。

61. Static Analysis｜靜態分析

不執行程式就分析風險。

工作原理：檢查安全漏洞、壞味道、死碼。

應用場景：AI review 後再跑掃描工具。

62. Smoke Test｜冒煙測試

快速確認基本功能能啟動。

工作原理：不用完整測試，只確認主要流程沒壞。

應用場景：部署後先檢查首頁、登入、主要按鈕。

63. E2E Test｜端對端測試

模擬使用者完整操作流程。

工作原理：從瀏覽器操作到後端資料完整驗證。

應用場景：Playwright/Cypress 測教材網站互動。

64. Test Fixture｜測試夾具

測試前準備的固定資料或環境。

工作原理：讓測試可重複、可預測。

應用場景：建立假使用者、假題庫、假回應。

65. Mock｜模擬物件

用假的依賴替代真服務。

工作原理：避免測試時真的呼叫 API 或資料庫。

應用場景：模擬 OpenAI API 回傳。

66. Assertion｜斷言

測試中用來判斷結果的條件。

工作原理：如果結果不符合預期，測試失敗。

應用場景：expect(total).toBe(100)。

67. Coverage｜測試覆蓋率

測試涵蓋程式碼的比例。

工作原理：可看出哪些檔案或分支沒被測到。

應用場景：AI 補測試時參考 coverage 報告。

68. Flaky Test｜不穩定測試

有時通過有時失敗的測試。

工作原理：常由時間、網路、非同步、外部依賴造成。

應用場景：AI 修 CI 失敗時要辨識是否 flaky。

69. Snapshot Test｜快照測試

保存輸出快照並比較差異。

工作原理：UI 或序列化輸出變動時會提醒。

應用場景：檢查名詞卡片 HTML 是否意外改變。

入門進階 2部署、環境與自動化

70. CI｜持續整合

每次提交後自動建置與測試。

工作原理：由 workflow 在 push 或 PR 時執行測試。

應用場景：AI 開 PR 後自動跑 lint/test/build。

71. CD｜持續部署

測試通過後自動部署。

工作原理：將合併後的程式發布到正式或測試環境。

應用場景：GitHub Pages、Netlify、Vercel 自動部署。

72. Workflow｜工作流程

自動化任務設定。

工作原理：由事件觸發 jobs，例如 push、PR、排程。

應用場景：GitHub Actions YAML。

73. Docker｜容器化工具

封裝應用與依賴的工具。

工作原理：用 image/container 建立一致環境。

應用場景：避免「我電腦可以跑，你電腦不行」。

74. Container｜容器

隔離執行的應用環境。

工作原理：容器包含執行所需檔案與依賴，彼此隔離。

應用場景：讓 AI 在容器中跑測試，降低主機風險。

75. Environment｜環境

程式執行的情境。

工作原理：常分 dev、staging、production。

應用場景：AI 修 bug 要確認錯誤在哪個環境發生。

76. Staging｜預備環境

正式上線前的測試環境。

工作原理：接近 production 但不影響真使用者。

應用場景：AI 修改先部署 staging 測試。

77. Production｜正式環境

真使用者正在使用的環境。

工作原理：穩定性、安全性與備份要求最高。

應用場景：AI 不應直接改 production。

78. Rollback｜回滾

把系統恢復到先前版本。

工作原理：部署失敗時快速切回穩定版本。

應用場景：AI 改壞網站時用 git 或平台還原。

79. Release｜發布版本

把功能正式交付使用。

工作原理：可包含版本號、release notes、tag。

應用場景：v1.2.0 新增搜尋與一鍵複製。

80. Semantic Versioning｜語意化版本

用 MAJOR.MINOR.PATCH 表示版本。

工作原理：破壞性變更、功能新增、修補分別升版。

應用場景：套件與教材工具版本管理。

81. Feature Flag｜功能旗標

用開關控制功能是否啟用。

工作原理：不重新部署也能開關新功能。

應用場景：先讓教師帳號試用新 AI 功能。

82. Observability｜可觀測性

了解系統內部狀態的能力。

工作原理：結合 log、metrics、traces。

應用場景：AI 修線上問題時要先看監控資料。

83. Monitoring｜監控

持續追蹤系統狀態。

工作原理：檢查錯誤率、回應時間、CPU、流量。

應用場景：網站掛掉時自動警示。

84. Incident｜事故

影響服務的異常事件。

工作原理：需要記錄時間線、影響、原因與修復。

應用場景：AI 部署造成網站不可用。

入門進階 3安全與風險

85. Sandbox｜沙盒

受限制的執行環境。

工作原理：限制檔案、網路、命令與權限。

應用場景：AI Agent 改程式時避免誤刪或執行危險指令。

86. Prompt Injection｜提示注入

把惡意指令藏在資料中誘導 AI。

工作原理：AI 可能把外部資料誤當指令執行。

應用場景：讀 README、網頁、email 前要有安全邊界。

87. Data Leakage｜資料外洩

敏感資料被不該看到的系統取得。

工作原理：可能經 prompt、log、API、工具回傳外流。

應用場景：學生個資、API key 不可貼入不可信工具。

88. Secret｜機密金鑰

API key、token、密碼等。

工作原理：應放環境變數或 secret manager，不可寫死。

應用場景：OPENAI_API_KEY、SUPABASE_SERVICE_ROLE_KEY。

89. Permission｜權限

AI 或工具可做的操作範圍。

工作原理：最小權限原則能降低失控影響。

應用場景：允許讀檔與測試，但不直接部署或刪庫。

90. Supply Chain Risk｜供應鏈風險

第三方套件與工具帶來的風險。

工作原理：npm 套件、GitHub Action、MCP server 可能被污染。

應用場景：AI 建議安裝套件前要查來源與權限。

91. Indirect Prompt Injection｜間接提示注入

惡意指令藏在外部資料中。

工作原理：AI 讀網頁、文件、README 時被資料操控。

應用場景：Coding Agent 讀不可信 repo 特別危險。

92. Capability｜能力範圍

工具或 Agent 被授權可做的事。

工作原理：包含讀檔、寫檔、網路、執行命令、發送請求。

應用場景：MCP server 應清楚宣告與限制能力。

93. Least Privilege｜最小權限

只給完成任務必需的權限。

工作原理：即使 AI 被誘導，也降低傷害。

應用場景：只開專案資料夾，不開整個硬碟。

94. Approval Gate｜審核閘門

高風險操作前的人工確認點。

工作原理：刪除、部署、寄送、付費、外部請求需確認。

應用場景：AI 自動化流程必備。

95. Allowlist｜允許清單

只允許指定項目通過。

工作原理：比黑名單更容易控管風險。

應用場景：只允許 AI 執行 npm test、npm run build。

96. Denylist｜拒絕清單

禁止特定命令或資源。

工作原理：阻擋 rm -rf、curl pipe shell 等危險模式。

應用場景：AI 終端機安全策略。

97. Provenance｜來源脈絡

資料或決策從哪裡來。

工作原理：追蹤來源可判斷可信度與責任。

應用場景：AI 引用文件、工具回傳、外部網頁時標示來源。

98. Audit Log｜稽核紀錄

記錄誰在何時做了什麼。

工作原理：用於追蹤安全事件與錯誤操作。

應用場景：AI Agent 操作檔案、API、部署都要留紀錄。

99. PII｜個人可識別資訊

可識別個人的資料。

工作原理：姓名、email、電話、學生資料都可能是 PII。

應用場景：餵 AI 前要遮蔽學生姓名與聯絡資訊。

100. Threat Model｜威脅模型

系統可能遭遇的攻擊與風險分析。

工作原理：列出資產、攻擊者、入口、影響與防護。

應用場景：設計 AI Agent 前先想誰能誘導它做壞事。

入門進階 4工程工作流

101. Refactor｜重構

不改功能但改善結構。

工作原理：整理命名、拆模組、移除重複、降低耦合。

應用場景：AI 很適合做小步重構，但要測試保護。

102. Boilerplate｜樣板程式碼

專案常見起始架構。

工作原理：包含初始化設定、目錄、範例元件。

應用場景：快速建立 React、Express、HTML 工具。

103. Scaffold｜鷹架

建立專案骨架。

工作原理：先生成資料夾、路由、元件與設定檔。

應用場景：讓 AI 先搭自然科互動教材網站骨架。

104. Code Review｜程式碼審查

檢查程式正確性、安全與維護性。

工作原理：審查 diff、架構、測試、風險與風格。

應用場景：AI 改完後請另一個 AI 或人類 review。

105. Technical Debt｜技術債

快速開發留下的結構問題。

工作原理：短期省時間，長期增加維護成本。

應用場景：Vibe Coding 連續補丁容易累積技術債。

106. Minimal Reproducible Example｜最小可重現範例

能重現 bug 的最小程式。

工作原理：排除無關因素，只保留錯誤核心條件。

應用場景：向 AI 提問時提供最小範例，答案更準。

入門進階 5需求與產品定義

107. User Story｜使用者故事

用使用者角度描述需求。

工作原理：通常寫成：身為某角色，我想做某事，以便達成某目標。

應用場景：把模糊需求轉成 AI 可拆解的功能。

108. Acceptance Criteria｜驗收條件

判斷功能是否完成的標準。

工作原理：用可測試條件定義完成狀態。

應用場景：避免 AI 只做出看起來像完成的原型。

109. Specification / Spec｜規格書

功能、限制、資料、流程的明確描述。

工作原理：把需求從聊天語言轉成工程規格。

應用場景：交給 AI 前先寫 spec，能降低返工。

110. PRD｜產品需求文件

產品層級的需求說明。

工作原理：包含目標、使用者、功能、限制、成功指標。

應用場景：大型 AI 專案開始前先產生 PRD。

111. Scope｜範圍

這次任務要做與不做的邊界。

工作原理：明確界定檔案、功能、時間與風險。

應用場景：要求 AI 不碰登入、不改資料庫、不重寫樣式。

112. Out of Scope｜不在範圍內

明確排除的項目。

工作原理：避免 AI 順手多做造成破壞。

應用場景：本次只修匯入，不重新設計 UI。

113. MVP｜最小可行產品

能驗證核心價值的最小版本。

工作原理：先做最少功能確認方向，再逐步擴充。

應用場景：用 AI 快速做教學工具原型。

114. Spike｜技術探勘

短時間研究可行性的任務。

工作原理：先驗證方案、套件、API 是否能用。

應用場景：請 AI 比較三種 PDF 轉 HTML 方法。

115. Backlog｜待辦清單

尚未完成的需求集合。

工作原理：可依優先級、難度、風險排序。

應用場景：AI 可幫你把聊天需求整理成 backlog。

116. Roadmap｜路線圖

專案功能演進計畫。

工作原理：分階段安排版本、里程碑與依賴。

應用場景：把教材平台規劃成 v1/v2/v3。

入門進階 6架構與設計

117. Architecture｜系統架構

系統元件與關係的整體設計。

工作原理：決定前端、後端、資料庫、API 與部署方式。

應用場景：讓 AI 先畫架構再寫程式。

118. System Design｜系統設計

從需求推導技術方案。

工作原理：考慮資料流、容量、安全、可靠性與維護性。

應用場景：避免 AI 只產生單檔式玩具程式。

119. Design Doc｜設計文件

描述技術決策與替代方案。

工作原理：記錄為什麼這樣設計、取捨與風險。

應用場景：讓 AI 提案前先產生 design doc。

120. ADR｜Architecture Decision Record

架構決策紀錄。

工作原理：用短文件記錄某次重要技術選擇。

應用場景：為什麼選 Supabase 而不是 Firebase。

121. Module｜模組

可獨立理解與維護的程式單元。

工作原理：把功能拆成邊界清楚的檔案或套件。

應用場景：AI 重構時將搜尋、匯出、UI 拆模組。

122. Component｜元件

可重複使用的 UI 或功能單位。

工作原理：前端框架中常用 props/state 組合畫面。

應用場景：按鈕、名詞卡、搜尋框、複製區塊。

123. API｜應用程式介面

系統之間交換資料的約定。

工作原理：定義路徑、方法、參數、回傳與錯誤。

應用場景：AI 連接前端與後端時必須先確認 API。

124. Endpoint｜端點

API 的具體網址或方法。

工作原理：例如 GET /terms、POST /feedback。

應用場景：除錯時確認 AI 呼叫的 endpoint 是否正確。

125. Payload｜資料負載

請求或回應中傳送的資料。

工作原理：通常是 JSON body、表單或檔案。

應用場景：AI 產生 API 測試時要檢查 payload 格式。

126. State｜狀態

程式當下記錄的資料。

工作原理：UI、使用者、表單、載入狀態都可能是 state。

應用場景：互動教材中目前題目、分數、進度。

127. Stateless｜無狀態

每次請求不依賴伺服器記憶。

工作原理：狀態放在 client、token 或資料庫。

應用場景：API 設計、serverless function 常見。

128. Stateful｜有狀態

系統會保存前後關係。

工作原理：session、連線、長任務進度都屬於狀態。

應用場景：聊天機器人、任務佇列、WebSocket。

入門進階 7工具與終端機

129. CLI｜命令列介面

用文字指令操作工具。

工作原理：透過 terminal 執行安裝、測試、建置、部署。

應用場景：Codex CLI、Git、npm、Docker。

130. Shell Command｜終端機指令

在作業系統中執行的命令。

工作原理：可讀寫檔案、安裝套件、啟動服務。

應用場景：AI 執行前要審查高風險命令。

131. Working Directory｜工作目錄

目前指令執行所在資料夾。

工作原理：相對路徑會依工作目錄解讀。

應用場景：很多 AI 失誤來自跑錯資料夾。

132. Environment Variable｜環境變數

系統層級的設定值。

工作原理：程式可從環境讀取 key、mode、URL。

應用場景：存 OPENAI_API_KEY、DATABASE_URL。

133. Package Manager｜套件管理器

安裝與管理依賴的工具。

工作原理：npm、pnpm、pip、uv、poetry 等。

應用場景：AI 新增套件前要確認必要性。

134. Dependency｜依賴套件

專案需要的外部程式庫。

工作原理：記錄在 package.json、requirements.txt 等。

應用場景：更新依賴可能修 bug，也可能造成破壞。

135. Lockfile｜鎖定檔

固定依賴版本的檔案。

工作原理：確保不同環境安裝相同版本。

應用場景：package-lock.json、pnpm-lock.yaml、poetry.lock。

136. Build｜建置

把原始碼轉成可執行或可部署成果。

工作原理：編譯、打包、壓縮、產生靜態檔。

應用場景：npm run build、vite build。

137. Dev Server｜開發伺服器

本機預覽與熱更新服務。

工作原理：監看檔案變更並即時刷新。

應用場景：AI 修改前端後用 localhost 檢查。

138. Log｜日誌

程式執行紀錄。

工作原理：記錄錯誤、請求、狀態與診斷資訊。

應用場景：貼 log 給 AI 可快速除錯。

入門進階 8資料、API 與後端

139. Database｜資料庫

保存結構化資料的系統。

工作原理：用表格、文件、鍵值或圖結構儲存。

應用場景：使用者、文章、回饋、檢核表資料。

140. Schema｜資料結構

資料庫或資料格式的欄位規則。

工作原理：定義欄位、型別、關聯與限制。

應用場景：Supabase 表格設計。

141. Migration｜資料庫遷移

資料庫結構的版本化變更。

工作原理：用腳本新增欄位、表格、索引。

應用場景：AI 改資料庫前必須產生 migration。

142. ORM｜物件關聯映射

用程式物件操作資料庫。

工作原理：把 SQL 表轉成程式語言中的物件。

應用場景：Prisma、SQLAlchemy、Django ORM。

143. REST API｜REST 介面

以 HTTP 方法操作資源的 API。

工作原理：GET/POST/PUT/DELETE 對應讀寫刪改。

應用場景：教材、題庫、活動資料 API。

144. GraphQL｜圖形查詢語言

由 client 指定需要哪些資料。

工作原理：單一 endpoint，查詢結構由 query 決定。

應用場景：避免前端拿太多或太少資料。

145. Webhook｜事件回呼

某事件發生時自動通知外部系統。

工作原理：服務端把事件資料 POST 到指定網址。

應用場景：GitHub PR 更新後觸發自動檢查。

146. Rate Limit｜速率限制

限制一段時間內請求數。

工作原理：防止濫用、保護服務穩定與成本。

應用場景：AI 大量呼叫 API 時容易撞限制。

147. Pagination｜分頁

分批取得大量資料。

工作原理：用 page、cursor、limit 控制回傳範圍。

應用場景：AI 搜 GitHub issue 不能一次抓全部。

148. Idempotency｜冪等性

重複執行結果仍相同。

工作原理：同一請求重送不應造成重複副作用。

應用場景：AI 失敗重試寄信、付款、建立事件時很重要。

入門進階 9前端與互動介面

149. DOM｜文件物件模型

瀏覽器中的 HTML 結構樹。

工作原理：JavaScript 透過 DOM 讀寫畫面元素。

應用場景：AI 修一鍵複製、按鈕、搜尋互動。

150. Event Listener｜事件監聽器

等待使用者或系統事件。

工作原理：按下、輸入、滑動、載入時觸發函式。

應用場景：複製按鈕、篩選按鈕、表單送出。

151. RWD｜響應式網頁設計

不同螢幕都能良好閱讀。

工作原理：用 media query、彈性排版、相對尺寸。

應用場景：技術文章可在手機、平板、桌機閱讀。

152. Accessibility / A11y｜無障礙設計

讓更多使用者能操作網頁。

工作原理：考慮對比、鍵盤、標籤、語意 HTML。

應用場景：AI 產生 UI 後要檢查 aria-label 與標題結構。

153. Hydration｜水合

前端接手伺服器產生的 HTML。

工作原理：SSR 頁面載入後由 JS 綁定互動。

應用場景：Next.js / Nuxt 常見錯誤來源。

154. SSR｜伺服器端渲染

在伺服器產生 HTML。

工作原理：提升首屏速度與 SEO，但環境更複雜。

應用場景：內容網站、文件站、SEO 頁面。

155. CSR｜客戶端渲染

在瀏覽器用 JavaScript 產生畫面。

工作原理：初始 HTML 較少，互動性強。

應用場景：單頁應用、儀表板、互動教材。

156. SPA｜單頁應用

不重整頁面切換畫面的應用。

工作原理：路由在前端控制，資料透過 API 取得。

應用場景：互動式教材平台與後台管理系統。

三、實戰流程：由入門到進階的 AI Vibe Coding 工作法

階段核心問題關鍵名詞產出物
1. 需求釐清到底要解決誰的什麼問題？User Story、Acceptance Criteria、Scope、MVPPRD、Spec、Backlog
2. 架構設計要用什麼技術、資料如何流動？Architecture、API、Schema、ADRDesign Doc、架構圖
3. AI 執行AI 可以看到什麼、能做什麼？Context Engineering、Tool Calling、Agent Loop任務計畫、修改檔案
4. 測試驗證如何證明沒有壞？Unit Test、E2E、Lint、Type Checking測試報告、修正紀錄
5. 審查部署誰檢查？如何回滾？PR、Code Review、CI/CD、RollbackPR、Release Note
6. 安全維護如何避免失控與外洩？Sandbox、Least Privilege、Audit Log、Prompt Injection權限設定、安全規則

四、進階篇：AI Vibe Coding 工程師必懂術語

這一區聚焦「把 AI 從聊天工具升級成工程協作者」所需的概念：上下文工程、規格鎖定、Agent 協作、工具整合、部署維運與評估測試。

157. Context Budget｜上下文預算

規劃一次任務中可放入模型的有效資訊量。

工作原理：把需求、檔案片段、錯誤訊息、文件與歷史紀錄做取捨，避免塞太多無關內容。

應用場景：大型專案請 AI 讀檔前，先要求它列出需要的檔案，而不是整包丟進去。

風險提醒：上下文不足會亂猜；上下文太滿會讓重點被稀釋。

158. Context Compression｜上下文壓縮

把長對話或大量檔案摘要成保留決策與約束的短版脈絡。

工作原理：用摘要、索引、任務狀態、決策紀錄取代完整歷史。

應用場景：長任務做到一半時，請 AI 產生 handoff note 給下一輪接續。

風險提醒：壓縮錯誤會遺失關鍵限制，需保留已確認決策。

159. Context Index｜上下文索引

幫大型專案建立檔案與知識定位表。

工作原理：先掃描路徑與責任，再依任務只讀必要區域。

應用場景：AI 修改 Science Portal 前，先建立 components、pages、config、data 的索引。

風險提醒：索引過舊會導致 AI 找錯檔案。

160. Context Drift｜上下文漂移

任務進行中，AI 對原始需求的理解逐漸偏離。

工作原理：長對話、多次修正、錯誤假設都可能讓目標變形。

應用場景：每幾輪要求 AI 重述目前任務、不可改項目與完成標準。

風險提醒：容易造成越改越遠、修 A 壞 B。

161. Spec Lock｜規格鎖定

在開發前凍結本輪要遵守的需求邊界。

工作原理：把功能範圍、不可更動項、驗收條件寫成明確清單。

應用場景：請 AI 改 HTML 時，鎖定文字內容不變，只調整樣式與互動。

風險提醒：規格沒鎖定，AI 可能自作主張。

162. Acceptance Test｜驗收測試

用使用者角度確認功能是否完成。

工作原理：把需求轉成可觀察的行為：點擊、輸入、顯示、儲存、錯誤處理。

應用場景：一鍵複製按鈕要能複製、提示成功、失敗時不讓頁面當掉。

風險提醒：只跑單元測試不代表使用者流程正常。

163. Definition of Done｜完成定義

明確定義什麼情況下任務才算完成。

工作原理：包含功能、測試、文件、審查、部署與回滾準備。

應用場景：AI 修 bug 後必須提供修改檔案、測試指令、結果與風險。

風險提醒：沒有 DoD，AI 可能只完成表面功能。

164. Non-functional Requirement｜非功能性需求

功能之外的品質條件。

工作原理：包含效能、安全、可用性、可維護性、可讀性、無障礙。

應用場景：技術手冊不只要內容完整，也要手機可讀、可列印、可搜尋。

風險提醒：常被 AI 忽略，但影響實際可用性。

165. Constraint Prompting｜限制式提示

在 prompt 中明確寫出不能做什麼。

工作原理：限制檔案範圍、套件新增、重寫行為、外部請求與部署操作。

應用場景：請 AI 只修改 CSS，不動 HTML 內容與 JS 邏輯。

風險提醒：限制不清會提高副作用。

166. Negative Prompting｜反向提示

明確排除不想要的輸出或行為。

工作原理：用禁止事項降低模型走錯方向的機率。

應用場景：不要加入行銷語、不要改標題、不要產生不存在的參考文獻。

風險提醒：過度負向限制可能讓模型不敢完成必要修改。

167. Rubric Prompt｜評分規準提示

先制定評分規準，再讓 AI 產出或審查內容。

工作原理：把品質拆成明確項目與分數或等級。

應用場景：評估 AI 產生的教案、網站、程式碼與技術文件。

風險提醒：規準不合理會讓評估失真。

168. Critic Pass｜批判回合

讓 AI 以審查者身份檢查前一版輸出。

工作原理：先產出，再切換角色進行 correctness、risk、clarity 檢查。

應用場景：生成 HTML 後，再請 AI 專門檢查連結、按鈕、RWD 與錯字。

風險提醒：同一模型自評仍可能看漏，需要測試佐證。

169. Self-Reflection Loop｜自我反思迴圈

AI 完成任務後自行檢查是否符合目標。

工作原理：比較需求、輸出、測試結果與未完成項。

應用場景：讓 Agent 每次修改後回答：我改了什麼、沒改什麼、如何驗證。

風險提醒：不能把自我反思當成真實測試。

170. Planner-Executor Pattern｜規劃者－執行者模式

把任務拆成規劃與執行兩個角色。

工作原理：Planner 拆步驟，Executor 逐步改檔、跑測試、回報。

應用場景：大型改版先規劃資訊架構，再逐段生成 HTML。

風險提醒：Planner 錯了，Executor 會高效率做錯事。

171. Reviewer Agent｜審查代理

專門檢查其他 Agent 產出的結果。

工作原理：可針對安全、測試、文件、效能或程式風格審查。

應用場景：Codex 改完後，用 Reviewer Agent 審 PR。

風險提醒：Reviewer 不能取代人類責任。

172. Handoff｜交接

把任務從一個 Agent 或一輪對話交給下一個。

工作原理：交接內容應包含目標、狀態、已完成、待辦、風險與重要檔案。

應用場景：長時間生成教材、網站或手冊時避免斷線後重來。

風險提醒：交接不完整會造成重工。

173. Checkpoint｜檢查點

在關鍵階段保存狀態或決策。

工作原理：用 commit、summary、測試結果或備份檔記錄可回復狀態。

應用場景：AI 大改網站前先建立 checkpoint。

風險提醒：沒有 checkpoint，失敗時很難回滾。

174. Plan-and-Execute｜先規劃後執行

先產生步驟計畫，再逐步操作。

工作原理：降低 AI 一次性亂改的風險。

應用場景：要求 AI：先列出檔案與步驟，確認後再修改。

風險提醒：計畫可能過度複雜，需要限制範圍。

175. ReAct Pattern｜推理＋行動模式

讓 Agent 在思考、使用工具、觀察結果之間循環。

工作原理：每一步依工具回傳結果修正下一步。

應用場景：除錯時先查錯誤，再讀檔，再跑測試。

風險提醒：工具結果若被污染，後續推理也會被污染。

176. Tool Router｜工具路由器

決定任務應該呼叫哪個工具。

工作原理：根據使用者意圖、資料來源、權限與成本選擇搜尋、檔案、終端機或 API。

應用場景：AI 判斷要用 GitHub 查 PR，還是用本機 grep 找程式。

風險提醒：錯誤路由會造成資料不完整或權限風險。

177. Tool Result Validation｜工具結果驗證

檢查工具回傳內容是否可信與符合格式。

工作原理：驗證 schema、狀態碼、來源、時間與錯誤訊息。

應用場景：AI 搜到資料後，要檢查是否過期或非官方。

風險提醒：不驗證會把錯誤工具結果當真。

178. Idempotent Tool｜冪等工具

重複呼叫也不會造成額外副作用的工具。

工作原理：同一輸入多次執行結果一致，適合 Agent 重試。

應用場景：建立草稿可重試；直接寄信或付款通常不是安全冪等。

風險提醒：非冪等操作要加確認與唯一鍵。

179. Dry Run｜試跑

先模擬操作，不真正改動系統。

工作原理：顯示將修改的檔案、命令、SQL 或部署步驟。

應用場景：AI 執行資料庫 migration 前先 dry run。

風險提醒：不是所有工具的 dry run 都完全等同真實執行。

180. Rollback Plan｜回滾計畫

修改失敗時恢復原狀的方案。

工作原理：包含備份、前一版 commit、資料庫回滾、部署回退。

應用場景：AI 升級網站前先說明如何回到上一版。

風險提醒：沒有回滾就不適合做高風險變更。

181. Feature Flag｜功能旗標

用開關控制功能是否啟用。

工作原理：新功能先部署但只對部分人開啟。

應用場景：新版 AI 搜尋先只給測試帳號使用。

風險提醒：旗標太多會形成維護負擔。

182. Canary Release｜金絲雀釋出

先讓少數流量使用新版，再逐步擴大。

工作原理：觀察錯誤率、效能與使用者回饋。

應用場景：AI Agent 自動修復功能先在測試專案啟用。

風險提醒：監控不足時金絲雀也可能放大事故。

183. Blue-Green Deployment｜藍綠部署

同時維持新舊兩套環境，切換流量完成部署。

工作原理：藍色代表現行版，綠色代表新版；出事可切回。

應用場景：大型教材平台改版時降低中斷。

風險提醒：成本較高，需要資料同步策略。

184. Observability｜可觀測性

透過 logs、metrics、traces 理解系統狀態。

工作原理：不是只知道壞了，而是知道哪裡、為什麼、影響多大。

應用場景：AI 生成後端 API 後加入錯誤紀錄與請求追蹤。

風險提醒：過度收集可能暴露個資或增加成本。

185. Telemetry｜遙測資料

系統自動回傳的使用與運行資料。

工作原理：包含錯誤、效能、事件、使用次數。

應用場景：分析哪個 Prompt 模板最常被複製。

風險提醒：要避免收集敏感內容。

186. Trace｜追蹤紀錄

記錄一次請求或 Agent 任務的完整路徑。

工作原理：串起輸入、工具呼叫、外部 API、輸出與耗時。

應用場景：除錯 Agent 為何修改錯檔案。

風險提醒：Trace 可能包含機密，需控管存取。

187. SLO｜服務水準目標

系統可用性或效能的目標值。

工作原理：例如 99.9% 可用、API 95% 請求低於 500ms。

應用場景：AI 教材網站設定可用性與載入速度目標。

風險提醒：目標過高會增加成本。

188. SLA｜服務水準協議

對外承諾的服務品質條款。

工作原理：通常比 SLO 更正式，可能涉及責任與補償。

應用場景：提供校內 AI 工具給多人使用時要說清楚維護責任。

風險提醒：不要承諾自己無法維持的可用性。

189. Incident Response｜事故應變

系統故障或安全事件的處理流程。

工作原理：包含偵測、通報、止血、修復、復盤。

應用場景：AI Agent 誤刪資料或部署壞網站後的處理。

風險提醒：沒有演練時，事故中容易慌亂。

190. Postmortem｜事後檢討

事故後分析原因與改善措施。

工作原理：重點不是找戰犯，而是修流程與防再發。

應用場景：記錄 AI 錯改檔案的原因、缺少的測試與新的防線。

風險提醒：若只檢討人，不改系統，下次仍會發生。

191. Prompt Regression｜提示詞回歸

新版 Prompt 讓原本會成功的任務變差。

工作原理：用固定測試題比較不同 prompt 版本。

應用場景：改寫 AI 教案 prompt 後檢查是否仍符合 108 課綱格式。

風險提醒：Prompt 也需要版本控制。

192. Golden Dataset｜黃金測試集

人工確認正確的一組標準測試案例。

工作原理：用來評估 AI 輸出是否穩定、正確、符合格式。

應用場景：建立 20 個標準任務測試 AI Coding Agent。

風險提醒：資料集太小會無法代表真實情境。

193. Synthetic Test Case｜合成測試案例

由人或 AI 產生的測試案例。

工作原理：補足真實資料不足，測邊界與錯誤情境。

應用場景：產生各種 HTML 壞格式檢查轉檔工具。

風險提醒：合成案例可能太理想化。

194. Mutation Testing｜突變測試

故意改壞程式，看測試是否抓得出來。

工作原理：若測試抓不到突變，代表測試不夠強。

應用場景：檢查 AI 生成的測試是否只是表面覆蓋。

風險提醒：成本較高，不一定每個專案都需要。

195. Flaky Test｜不穩定測試

同樣程式有時過、有時失敗的測試。

工作原理：常因時間、網路、非同步、外部依賴造成。

應用場景：AI 修 CI 時要辨別是真 bug 還是 flaky test。

風險提醒：忽略 flaky test 會降低團隊對測試的信任。

196. Test Oracle｜測試判準

用來判斷輸出是否正確的標準。

工作原理：可以是預期值、規則、人工評分或參考文件。

應用場景：評估 AI 產生的摘要是否忠於原文。

風險提醒：判準模糊會導致測試失效。

五、專家篇：Agentic Coding 架構、安全與治理術語

這一區偏向設計 AI Coding 平台、企業級 Agent 工作流、RAG 安全、成本控制、可靠性與架構治理。適合進階工程師、技術負責人與想打造 AI 開發系統的人。

197. Agent Runtime｜代理執行環境

負責管理 Agent 迴圈、工具呼叫、狀態與權限的執行層。

工作原理：把模型輸出轉為工具操作，並處理錯誤、重試、暫停與回報。

應用場景：建立校內 AI 助教或 Coding Agent 平台時的核心。

風險提醒：Runtime 權限過大會放大事故。

198. Agent State Machine｜代理狀態機

用明確狀態管理 Agent 任務流程。

工作原理：例如 idle、planning、acting、waiting approval、testing、done。

應用場景：讓長任務不會亂跳步驟。

風險提醒：狀態設計過複雜會難維護。

199. Policy Engine｜政策引擎

根據規則決定 AI 可以或不可以做什麼。

工作原理：檢查操作類型、資料敏感度、使用者角色、環境與風險。

應用場景：部署、刪檔、讀取機密前要求人工批准。

風險提醒：政策不完整會留下繞過路徑。

200. Guardrail｜護欄

限制 AI 輸入、輸出或行動的安全控制。

工作原理：可在模型前、模型後、工具前、工具後加入檢查。

應用場景：阻擋危險命令、機密外洩、錯誤格式。

風險提醒：護欄不能只靠模型自律。

201. Output Filtering｜輸出過濾

在回覆送出前檢查敏感資訊或危險內容。

工作原理：用規則、分類器、掃描器或程式碼檢查。

應用場景：防止 Agent 把 API key、系統提示詞或學生個資輸出。

風險提醒：過濾太寬會誤擋正常內容。

202. Input Sanitization｜輸入清理

處理使用者或外部資料中的危險內容。

工作原理：移除或標記不可信指令、HTML、腳本、控制字元。

應用場景：AI 讀網頁、email、issue 前先標示為 untrusted content。

風險提醒：清理不代表完全安全。

203. Trust Boundary｜信任邊界

區分可信與不可信資料或系統的界線。

工作原理：資料跨越邊界時必須驗證、授權、記錄。

應用場景：使用者輸入、GitHub issue、外部網頁都不應被當成系統指令。

風險提醒：邊界不清是 prompt injection 的溫床。

204. Capability Token｜能力權杖

只授予特定能力與範圍的存取憑證。

工作原理：比全域 API key 更細緻，限制時間、資源與動作。

應用場景：給 Agent 一次性讀檔權限，而不是整個雲端硬碟權限。

風險提醒：Token 管理複雜，需設定過期與撤銷。

205. Least Privilege｜最小權限

只給完成任務所需的最低權限。

工作原理：把讀、寫、刪除、部署、外部網路分開授權。

應用場景：AI 修 HTML 不需要資料庫管理員權限。

風險提醒：權限過小會降低效率，但比過大安全。

206. Audit Log｜稽核紀錄

記錄誰在何時做了什麼操作。

工作原理：包含工具呼叫、檔案修改、資料存取、批准紀錄。

應用場景：追查 AI Agent 為何改了某個檔案。

風險提醒：Log 本身也可能含敏感資訊。

207. Provenance｜來源溯源

追蹤資料、程式碼或回答的來源。

工作原理：記錄由誰產生、根據哪些文件、哪個版本、何時建立。

應用場景：技術文章引用官方文件與日期。

風險提醒：沒有 provenance，錯誤很難追責。

208. Data Lineage｜資料血緣

追蹤資料從來源到輸出的流動與轉換。

工作原理：記錄匯入、清理、切片、embedding、檢索與生成流程。

應用場景：RAG 知識庫更新後確認哪些回答受影響。

風險提醒：資料血緣缺失會造成舊資料混用。

209. PII Detection｜個資偵測

辨識姓名、電話、email、地址、學號等個資。

工作原理：用規則、模型或 DLP 工具掃描。

應用場景：學生資料進入 AI 前先遮罩。

風險提醒：中文姓名與在地格式需要客製規則。

210. Redaction｜遮蔽處理

移除或遮住敏感資料。

工作原理：把真實值換成代碼、星號或假資料。

應用場景：學生姓名中間改成全形Ｏ。

風險提醒：遮蔽後仍可能透過上下文推回身份。

211. Data Retention｜資料保留政策

規定資料保存多久與何時刪除。

工作原理：依資料類型、法規、用途與風險設定期限。

應用場景：AI 工具的 log 與上傳檔案不應無限期保存。

風險提醒：刪除政策若無法執行就只是口號。

212. Vector Poisoning｜向量資料污染

惡意或錯誤文件進入向量庫，影響檢索結果。

工作原理：污染內容可能被檢索並誤導模型回答或行動。

應用場景：不可信 README 藏入惡意 Agent 指令。

風險提醒：RAG 文件要有審核、來源與隔離。

213. Embedding Drift｜嵌入漂移

模型或資料更新後，向量語意分布改變。

工作原理：新舊 embedding 混用可能導致搜尋品質下降。

應用場景：更換 embedding model 後重新建立索引。

風險提醒：未重建索引會造成檢索不穩。

214. Hybrid Search｜混合搜尋

結合關鍵字與向量語意搜尋。

工作原理：BM25 找精確詞，embedding 找語意相近內容，再合併排序。

應用場景：搜尋專有名詞、錯字、中文術語時更穩定。

風險提醒：合併權重需要調校。

215. Reranker｜重排序器

對初步檢索結果重新排序。

工作原理：用更精細模型判斷文件與問題的相關性。

應用場景：RAG 回答前從 20 段候選中選最相關 5 段。

風險提醒：增加延遲與成本。

216. Query Expansion｜查詢擴展

把使用者問題改寫成多個搜尋查詢。

工作原理：加入同義詞、英文術語、縮寫、相關概念。

應用場景：搜尋 Vibe Coding 同時查 agentic coding、AI coding workflow。

風險提醒：擴展太多會引入噪音。

217. Citation Grounding｜引用依據化

要求回答中的關鍵說法能對應到來源。

工作原理：把生成內容與檢索片段建立可追蹤關係。

應用場景：技術手冊中每個新標準引用官方文件。

風險提醒：引用不等於理解，仍需檢查是否支持該說法。

218. Semantic Cache｜語意快取

把相似問題的回答或檢索結果暫存。

工作原理：用 embedding 判斷新問題是否可重用舊結果。

應用場景：常見 AI Coding 名詞查詢可快速回覆。

風險提醒：快取過期會回傳舊知識。

219. Cost Guardrail｜成本護欄

限制模型與工具使用成本。

工作原理：設定 token、請求次數、工具呼叫、重試與時間上限。

應用場景：Agent 卡住時自動停止，避免無限燒錢。

風險提醒：限制太緊可能中斷正常任務。

220. Latency Budget｜延遲預算

規劃一次使用者請求可接受的最長等待時間。

工作原理：把模型、檢索、工具、資料庫與前端回應都納入。

應用場景：即時教學助理要比離線報告更重視延遲。

風險提醒：太追求速度可能犧牲品質。

221. Token Accounting｜Token 成本核算

追蹤輸入、輸出與工具資料造成的 token 消耗。

工作原理：用於成本估算、限制、優化 prompt 與上下文。

應用場景：大型文件摘要前先估算 token。

風險提醒：只看 token 不看工具成本仍不完整。

222. Batching｜批次處理

把多個小任務合併處理。

工作原理：降低固定開銷，提高吞吐量。

應用場景：一次處理多篇教材摘要或多個檔案檢查。

風險提醒：批次太大會延遲單一任務。

223. Queue｜佇列

排隊處理任務的機制。

工作原理：高峰時把工作放進 queue，worker 依序消化。

應用場景：大量文件轉 embedding、批次產生 HTML。

風險提醒：佇列失敗需要重試與死信處理。

224. Dead Letter Queue｜死信佇列

存放多次處理失敗的任務。

工作原理：避免壞任務卡住整個系統。

應用場景：某份 PDF 解析失敗後移到人工檢查。

風險提醒：不處理死信會累積隱性問題。

225. Circuit Breaker｜斷路器

外部服務連續失敗時暫停呼叫。

工作原理：防止系統不斷重試拖垮自己或對方。

應用場景：AI 搜尋 API 故障時暫停，改用快取或降級。

風險提醒：恢復策略要設計好。

226. Backoff｜退避重試

失敗後逐漸延長重試間隔。

工作原理：常用 exponential backoff 避免瞬間大量重試。

應用場景：API rate limit 或網路不穩時使用。

風險提醒：對非冪等操作重試要小心。

227. Timeout｜逾時

超過指定時間就停止等待。

工作原理：避免工具、API 或 Agent 任務無限卡住。

應用場景：AI 執行 npm install 或測試時設定合理逾時。

風險提醒：逾時太短會誤殺正常任務。

228. Concurrency Control｜併發控制

管理多個任務同時操作共享資源。

工作原理：用鎖、交易、版本號避免互相覆蓋。

應用場景：兩個 Agent 不應同時改同一份設定檔。

風險提醒：控制不足會造成資料競爭。

229. Optimistic Locking｜樂觀鎖

更新時檢查版本是否仍一致。

工作原理：假設衝突少，最後寫入前比對版本。

應用場景：多人或多 Agent 同時編輯文件時避免覆蓋。

風險提醒：衝突處理要清楚。

230. Schema Evolution｜結構演進

資料結構隨版本逐步變更。

工作原理：透過 migration、向後相容與版本標記維持穩定。

應用場景：教材 metadata 欄位增加難度、年級、課綱代碼。

風險提醒：不相容變更會破壞舊資料。

231. API Versioning｜API 版本控管

管理 API 介面變更。

工作原理：用 /v1、/v2、header 或相容策略支援不同客戶端。

應用場景：AI 產生前端時要確認呼叫哪個 API 版本。

風險提醒：沒有版本控管會讓舊前端突然壞掉。

232. Contract Testing｜契約測試

測試服務之間約定的 API 格式。

工作原理：確認 provider 與 consumer 對欄位、型別、錯誤碼理解一致。

應用場景：前端與 Supabase Edge Function 之間的資料格式。

風險提醒：契約沒更新會造成測試誤報。

233. Monorepo｜單一儲存庫多專案

把多個相關套件或應用放在同一 repository。

工作原理：共享工具、型別、元件、設定與 CI。

應用場景：教材網站、後台、共用 UI 元件放一起。

風險提醒：規模變大後建置與權限管理更複雜。

234. Polyrepo｜多儲存庫架構

不同服務或套件分開 repository。

工作原理：獨立權限、版本、部署與生命週期。

應用場景：校內多個獨立小工具分開管理。

風險提醒：跨 repo 改動較難協調。

235. ADR｜架構決策紀錄

記錄重要技術選擇與理由。

工作原理：包含背景、選項、決策、後果。

應用場景：記錄為何選 Supabase 而不是 Firebase。

風險提醒：不寫 ADR，半年後沒人知道當初為何這樣設計。

236. Runbook｜維運手冊

事故或例行操作的標準步驟文件。

工作原理：寫清楚如何重啟、回滾、查 log、聯絡誰。

應用場景：GitHub Pages 壞掉時依 Runbook 檢查。

風險提醒：過期 Runbook 比沒有更危險。

六、Prompt 範本：入門、進階、專家可直接套用

專案健檢

請先閱讀專案結構，不要修改檔案。請列出：技術棧、入口檔、主要模組、測試方式、可能風險。

最小修改

請用最小修改修正問題。限制：不要重寫整個檔案、不要新增不必要套件、修改後列出檔案與測試結果。

Agent 安全執行

在執行任何 shell command 前，請先說明目的、風險與替代方案。遇到刪除、部署、外部請求、金鑰操作時必須先停下來等待確認。

Code Review

請審查這次 diff，從 correctness、security、maintainability、test coverage、regression risk 五個角度列出問題與建議。

RAG 設計

請為這批文件設計 RAG 知識庫，包含 chunk 策略、metadata、embedding、retrieval、grounding、更新與淘汰策略。

進階：上下文壓縮交接

請把目前任務壓縮成 handoff note，保留：目標、已完成、待辦、重要檔案、不可更動限制、測試結果、已知風險。不要加入推測。

進階：規格鎖定後再開發

在開始修改前，請先產生 Spec Lock：本輪目標、範圍內、範圍外、不可更動項、驗收測試。完成後再提出最小修改計畫。

專家：Agent 安全政策檢查

請在每個工具呼叫前進行 policy check：資料來源是否可信、是否涉及機密、是否會寫入/刪除/部署、是否需要人工批准、是否有 rollback plan。

專家：RAG 安全與引用檢查

請檢查這個 RAG 回答：每個關鍵說法是否有來源支持、是否混入過期資料、是否有向量資料污染風險、是否需要拒答或補查官方文件。

專家：部署前 Release Gate

請建立部署前 release gate 清單：測試、lint、build、資安掃描、環境變數、回滾計畫、監控指標、影響範圍、人工核准項目。
