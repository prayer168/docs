一、Cursor 是什麼？

Cursor 是一款以 AI Coding 為核心的程式碼編輯器，可以把它理解成「強化 AI Agent 能力的 VS Code 類工具」。它不只是自動補字，而是能讀取專案脈絡、協助修改多個檔案、解釋錯誤、撰寫測試、重構程式，甚至透過 Agent 模式完成一段開發任務。

Tab智慧補全根據上下文補完程式碼，適合日常撰寫函式、HTML、CSS、Python、JavaScript。
Chat問答與解釋可以針對目前檔案、選取程式碼或整個專案提問。
Agent多檔案任務可以要求 Cursor 分析專案、修改程式、建立檔案、執行測試並回報結果。
一句話理解： VS Code 比較像「專業編輯器」，Cursor 則更像「內建 AI 程式助理的編輯器」。新手可以用它學習，進階者可以用它加速開發，但仍需要人類負責需求判斷、架構決策、測試與驗收。

二、安裝前準備

建議先安裝的工具

工具用途建議
Git版本控制、下載 GitHub 專案、提交修改強烈建議安裝
Node.js LTS前端專案、Vite、React、Electron、各種 AI Coding 工具做網頁或桌面程式必備
Python資料分析、自動化、Streamlit、科學計算做教學工具或資料處理建議安裝

Windows 安裝流程

進入官方下載頁
到 Cursor 官方下載頁，選擇 Windows 版本。一般 Windows 11 電腦通常選 Windows x64 User 或 Windows x64 System。
User 與 System 的差異
User 安裝在目前使用者資料夾，通常不需要系統管理員權限；System 安裝給整台電腦所有使用者，通常需要管理員權限。
執行安裝檔
下載後執行安裝檔，依照精靈完成安裝。若 Windows Defender 或 SmartScreen 詢問，確認來源是官方網站後再允許。
開啟 Cursor
安裝完成後，從開始選單搜尋 Cursor，或在桌面捷徑開啟。

macOS 安裝流程

依照 Mac 晶片選擇 ARM64、x64 或 Universal。Apple Silicon 多半選 ARM64；Intel Mac 選 x64。
下載後打開 .dmg，將 Cursor 拖曳到 Applications。
第一次開啟若跳出安全性提醒，請到「系統設定 → 隱私權與安全性」允許開啟。

Linux 安裝流程

Linux 可依發行版選擇 .deb、RPM 或 AppImage。Ubuntu / Debian 系統通常選 .deb；Fedora / RHEL 系統通常選 RPM；想免安裝可試 AppImage。

Ubuntu / Debian 安裝 .deb 範例
sudo dpkg -i cursor*.deb
sudo apt -f install
版本提醒： 官方下載頁會持續更新版本。本文以 2026-07-09 查詢到的官方下載資訊撰寫，實際下載時請以官方頁面顯示為準。

三、首次開啟與基本設定

1. 登入帳號

第一次開啟 Cursor 後，通常會要求登入帳號。可以使用 Email、Google、GitHub 等方式登入。登入後才能使用 AI 功能與同步部分設定。

2. 匯入 VS Code 設定

如果原本使用 VS Code，建議匯入擴充套件、快捷鍵、佈景主題與設定。這樣可以降低轉換成本。

建議： 不要一次裝太多擴充套件。先保留 Git、Python、ESLint、Prettier、Live Server、Markdown、GitHub 相關工具即可。擴充套件太多時，AI Agent 可能遇到執行環境混亂或提示過多的問題。

3. 開啟專案資料夾

Cursor 最適合以「整個專案資料夾」為單位工作。不要只開單一檔案，因為 AI 需要讀取專案結構、package.json、README、設定檔與相關程式。

建議專案結構
my-project/
├─ README.md
├─ package.json
├─ src/
├─ public/
├─ docs/
└─ .cursor/
   └─ rules/

四、Cursor 基本使用方式

常用功能總覽

功能適合情境使用方式
Tab 補全寫函式、補 HTML/CSS、快速完成重複程式游標停在程式碼中，等待建議後按 Tab 接受
Inline Edit針對選取程式碼修改、解釋、重構選取程式碼後呼叫 AI 編輯指令
Chat問概念、除錯、請它分析目前檔案在聊天視窗輸入問題，可搭配 @ 檔案
Agent / Composer建立功能、修 bug、跨檔案修改描述任務、限制、驗收條件，讓 Agent 規劃後執行
@ 引用讓 AI 精準讀取特定檔案、資料夾、文件在 Prompt 中使用 @README.md、@src 等
終端機整合安裝套件、執行測試、啟動開發伺服器在 Cursor 內建 Terminal 執行 npm、python、git 等指令

常用快速鍵概念

不同版本與作業系統快捷鍵可能略有差異，建議到 Cursor 的 Command Palette 搜尋功能名稱。Windows / Linux 通常使用 Ctrl，macOS 通常使用 Command。

Command Palette搜尋所有指令、設定與功能，是新手找功能最快的入口。
Chat / Composer用自然語言提出任務，適合從「我要做什麼」開始，而不是從「我要改哪一行」開始。

五、AI Coding 建議工作流程

不要一開始就叫 Cursor「直接幫我做完整網站」。比較穩定的方式是把 AI 當成工程團隊助手，讓它先理解需求，再規劃、執行、測試、迭代。

需求說明
我要做什麼？給誰用？
專案盤點
目前有哪些檔案？
任務拆解
分成小步驟
執行修改
一次改一段
測試驗收
確認能跑、能用、能維護

推薦用法：先請 Cursor 讀專案

專案盤點 Prompt
請先不要修改任何檔案。請閱讀目前專案結構，整理這個專案的用途、主要技術、重要檔案、啟動方式，以及你建議我接下來優先檢查的地方。

推薦用法：先寫規格再開發

需求轉規格 Prompt
我想新增一個功能：〔在這裡描述功能〕。請先幫我寫一份簡短 PRD，包含使用者情境、功能需求、資料結構、頁面流程、驗收條件與可能風險。先不要改程式碼。

推薦用法：小步修改

小步修改 Prompt
請只完成第一階段：〔具體小任務〕。請列出你會修改哪些檔案，說明原因，完成後請提供測試方式。不要順手重構其他無關程式。

六、Cursor Rules：讓 AI 知道你的專案規範

Cursor Rules 可以理解成「專案內的 AI 工作守則」。當專案越來越大時，建議建立規範，避免 AI 每次都用不同風格、不同架構、不同命名。

建議規則內容

技術棧例如使用 React、Vite、Tailwind、Python、Streamlit。
命名規則檔名、函式、元件、資料夾如何命名。
修改原則不得任意刪除檔案、不得大幅重構、需保留註解。
輸出格式回覆需使用繁體中文、列出修改摘要與測試方式。
安全規範不得把 API Key 寫死在前端，不得提交 .env。
教學需求若是教材專案，需符合國小學生可讀性與 RWD。
Cursor Rules 範本
# Cursor Project Rules

## 語言與回覆
- 請使用繁體中文回覆。
- 修改程式後，請提供：修改摘要、受影響檔案、測試方式、可能風險。

## 開發原則
- 不要未經要求大幅重構。
- 不要刪除既有功能。
- 每次修改以最小可行變更為原則。
- 若需求不清楚，先提出假設並採用最保守做法。

## 安全原則
- 不要把 API Key、Token、密碼寫入前端或提交到 Git。
- 使用 .env 管理機密，並提供 .env.example。

## 教材專案原則
- 內容需適合國小與國中學生理解。
- 網頁需支援手機、平板與桌機。
- 重要操作要有清楚按鈕與提示文字。

七、Cursor CLI：在終端機使用 AI Agent

Cursor 除了桌面版，也提供 CLI，可用於終端機、自動化腳本、文件更新、安全檢查、CI/CD 流程等情境。官方 CLI 頁面提供安裝指令：

Cursor CLI 安裝指令
curl https://cursor.com/install -fsS | bash
Windows 提醒： 如果你在 Windows 使用 CLI，建議搭配 WSL / Ubuntu 或 Git Bash。若遇到 PATH 找不到指令，請重新開啟終端機，或檢查安裝訊息中提示的 PATH 設定。

CLI 適合做什麼？

情境範例
文件更新自動檢查 README、技術文件、CHANGELOG 是否與程式一致
安全檢查掃描是否有 API Key、危險套件、明顯漏洞
自動化任務在 GitHub Actions 或 CI 流程中要求 AI 檢查修改內容
大型專案輔助讓 Agent 在終端機環境中分析、修改、執行測試

八、常見踩雷與排除方式

問題可能原因解決方式
安裝後找不到 Cursor安裝在使用者資料夾、捷徑未建立從開始選單搜尋 Cursor；或到使用者 AppData / Applications 檢查安裝位置
Windows 出現安全性阻擋SmartScreen 或防毒軟體提醒確認下載來源是官方網站，再選擇允許執行
AI 無法理解專案只開單一檔案、缺 README、專案結構混亂用「Open Folder」開整個資料夾，補上 README 與規則檔
Agent 改太多檔案Prompt 太寬、沒有設定限制要求它先列修改計畫，並限制「只改指定檔案」
程式能跑但架構很亂只要求完成，沒有要求維護性與測試加上驗收條件、測試方式、不要重複程式碼、保持模組化
npm install 失敗Node 版本不合、套件衝突、網路問題安裝 Node.js LTS，刪除 node_modules 與 lock file 後重裝，或請 Cursor 分析錯誤訊息
Git 權限或推送失敗未登入 GitHub、SSH/Token 未設定先用 GitHub Desktop 或 gh CLI 完成登入，再回到 Cursor 操作
AI 產生假功能或亂引用沒有提供檔案脈絡或驗收標準使用 @ 引用檔案，要求「只根據目前專案內容回答」
最重要的踩雷提醒： 不要把 Cursor 當成「自動完成整個專案的神燈」。真正穩定的做法是：需求清楚、檔案脈絡清楚、修改範圍清楚、測試方式清楚。

九、Cursor 實用 Prompt 範本

1. 讀懂專案

讀懂專案
請先閱讀整個專案，不要修改檔案。請用繁體中文整理：
1. 專案用途
2. 技術棧
3. 啟動方式
4. 主要檔案
5. 目前可能的風險
6. 建議下一步。

2. 修 Bug

修 Bug
以下是錯誤訊息：〔貼上錯誤〕。
請先判斷可能原因，列出 3 個最可能的修正方向。
確認後再修改，不要一次改太多檔案。

3. 建立 HTML 技術文件

HTML 技術文件
請將以下內容整理成一份 RWD HTML 技術文件，包含目錄、章節卡片、表格、流程圖、常見問題、一鍵複製按鈕與參考資料。語氣使用繁體中文，適合教師與初學者閱讀。

4. 重構程式

重構
請檢查目前程式碼是否有重複、過長函式、命名不清楚、責任混雜等問題。
請先提出重構計畫，不要直接修改。
重構必須保持既有功能不變。

5. 產生測試

測試
請根據目前功能補上測試。
測試需涵蓋正常情境、邊界情境與錯誤情境。
完成後請告訴我如何執行測試，以及目前測試仍未涵蓋哪些風險。
