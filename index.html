後續實作會逐漸顯示，真正的知識庫不是「資料很多」，而是每一份資料都知道自己是什麼、從哪裡來、可以連到哪裡去，以及未來要被誰使用。

一、為什麼知識庫和業務數據庫要分家？

一開始用 Agent AI 的時候，很容易把所有東西混在一起：靈感、文章草稿、課程筆記、Prompt、Debug 紀錄、專案進度、行政資料、腳本設定、工具安裝紀錄，全部塞在同一個地方。短期看起來方便，長期就會變成一團難以維護的資訊泥巴。

層次放什麼目的
Obsidian Vault / LLM Wiki長期知識、概念、教學法、文章脈絡、寫作風格、可重用模板讓知識能沉澱
業務數據庫任務狀態、專案進度、行政資料、表格型資料、日期、金額、待辦事項讓資料能查詢
Agent 設定層CLAUDE.md、hooks、skills、memory routing、腳本入口讓流程能自動化
輸出層文章、簡報、HTML、PDF、教材包、發布紀錄讓成果能追溯
這樣分開之後，整個系統才會開始變清楚：知識要能沉澱，數據要能查詢，流程要能自動化，輸出要能追溯。

二、LLM Wiki 和一般筆記庫有什麼不同？

LLM Wiki 和一般筆記庫最大的差別在於：一般筆記庫主要是給人看的，LLM Wiki 則是要同時給人和 Agent 看的。

也就是說，它不只要能讀，還要能被理解、被引用、被更新、被串連。每一篇筆記最好都有標題、摘要、標籤、資料類型、更新時間、來源、相關概念，以及後續可以連結的內容。

YAML metadata 範例
---
title: "LLM Wiki 與 Obsidian Vault"
type: concept
status: active
created: 2026-07-06
updated: 2026-07-06
valid_until: 2026-12-31
source:
  - "https://obsidian.md/help/links"
related:
  - "[[Agent Memory]]"
  - "[[CLAUDE.md]]"
  - "[[Knowledge Graph]]"
---

這裡面最重要的是 type、status、valid_until 和 related。type 讓 Agent 知道資料類型，status 標示內容狀態，valid_until 避免舊資料假裝自己還是新的，related 則是 LLM Wiki 的靈魂。

三、新資料進來時，不只是新增，而是要接回知識網

在這樣的架構中，Agent 寫入新資料時，不只是新增一篇筆記，而是必須判斷資料類型、產生摘要與 metadata、找出相關概念、加入雙向連結、判斷是否需要回寫舊筆記，並留下來源與更新時間。

要求：
1. 使用繁體中文。
2. 加上 YAML frontmatter。

6. 若內容提到既有概念，請用 [[概念名稱]] 串接。
7. 最後列出「建議回寫到哪些舊筆記」。

四、CLAUDE.md、hooks 和 SKILL.md 要分工清楚

CLAUDE.md、hooks 和 SKILL.md 如果混在一起，Agent 很容易開始混亂。它可能把長期規則當成一次性任務，也可能把臨時紀錄當成永久知識。分清楚之後，Agent 才比較知道什麼該記住、什麼該執行、什麼該查詢、什麼該輸出。

機制適合放什麼不適合放什麼
CLAUDE.md每次都要知道的工作原則、專案架構、常用指令、寫作語氣、輸出規則、資料儲存路徑太長的教學文件、一次性任務
hooks必須自動執行的規則，例如寫入後產生紀錄、輸出前檢查格式、修改後更新 log需要大量理解與判斷的長文
SKILL.md可重複呼叫的專門流程，例如生成 HTML 技術文件、整理科展前置作業、轉換研習成果報告全域偏好、臨時備忘
Obsidian LLM Wiki長期知識、概念關係、歷程脈絡高頻變動的業務數據
業務數據庫任務狀態、表格資料、進度追蹤長篇概念文章

五、可直接放進 CLAUDE.md 的範例

CLAUDE.md 範例
# Project Memory

## Identity
- 使用繁體中文回應。
- 寫作風格要清楚、具體、有現場教師的實作感。
- 產出文章時，優先使用「技術文章 + 教育現場案例」的混合風格。

## Knowledge Routing
- 長期概念寫入 `Obsidian-Vault/01_Knowledge/`。
- 專案紀錄寫入 `Obsidian-Vault/02_Projects/`。
- 可重用模板寫入 `Obsidian-Vault/04_Templates/`。
- Agent 記憶與執行紀錄寫入 `Obsidian-Vault/05_Agent-Memory/`。
- 業務狀態、行政進度、表格資料不得寫入知識庫，應寫入業務數據庫。

## Writing Rules
- 新增知識筆記時，必須加入 YAML frontmatter。
- 每篇筆記至少加入 3 個 `[[雙向連結]]`。
- 若資料可能過期，必須加入 `valid_until`。
- 引用外部資料時，必須保留來源連結。

## Output Rules
- 技術文章需包含：背景、問題、架構、流程、範例、踩雷、參考文獻。
- 有指令或 Prompt 的地方，HTML 版需提供一鍵複製。

六、可直接放進 memory routing 的範例

Memory Routing Rules 範例
# Memory Routing Rules

## 判斷順序
1. 這是長期概念嗎？如果是，寫入 `01_Knowledge/`。
2. 這是某個專案的進度或決策嗎？如果是，寫入 `02_Projects/專案名稱/`。
3. 這是可重複使用的流程嗎？如果是，寫入 `04_Templates/` 或轉成 skill。
4. 這是錯誤、踩雷、修復歷程嗎？如果是，寫入 `05_Agent-Memory/Logs/`。
5. 這是行政、金額、日期、任務狀態等結構化資料嗎？如果是，寫入業務數據庫。

## 寫入要求
- 每次寫入都要產生 `summary`。
- 每次寫入都要補 `related`。
- 每次寫入都要判斷是否需要回寫舊筆記。
- 不確定分類時，先寫入 `00_Inbox/`，並標記 `status: needs-review`。

七、遷移會踩到的重點

遷移時最常見的問題，往往不是 Obsidian，也不是資料夾，而是「舊系統的路徑依賴」。

只要過去的 CLAUDE.md、hooks、memory、skills、scripts 有任何地方寫死路徑，搬家時就會一個一個浮出來。這也是為什麼真正要做 LLM Wiki，不能只看筆記本身，還要整理整個 Agent 工作環境。

  1. 路徑盡量集中在一個設定檔。
  2. 不要讓每個 skill 自己決定記憶要寫哪裡。
  3. 每次搬移資料夾後，要做一次連結檢查。
  4. CLAUDE.md 不要寫成百科全書，應該寫成導航與規則。
  5. 複雜流程要拆成 SKILL.md 或腳本。
  6. 每次大改架構，都要留下 migration log。
  7. 所有新資料都要有來源、類型、狀態與相關連結。

八、給剛開始使用 Agent AI 的人

對剛開始使用 Agent AI 的人來說，越早開始建立 Obsidian Vault + LLM Wiki 越好。

不要等到資料散落在桌面、Google Drive、聊天紀錄、GitHub、Notion、下載資料夾、各種臨時文件之後，才想辦法搬進去。那時候整理的成本會高很多，而且 Agent 也很難判斷哪些是重要記憶，哪些只是過期殘影。

  1. 建立固定的 Vault。
  2. 把知識、專案、模板、記憶、封存分開。
  3. 每篇筆記都加 metadata。
  4. 每次新增內容都加雙向連結。
  5. 把 Agent 的記憶寫入規則說清楚。
  6. 把業務數據和長期知識分開管理。

等這套系統慢慢長起來，你會發現 Agent 不只是幫你生成內容，而是能沿著你的知識脈絡繼續工作。它會知道一個概念從哪裡來，曾經跟哪些專案有關，哪些內容已經更新，哪些文章可以重組成新的教材、貼文、簡報或技術文件。

這才是 LLM Wiki 真正的價值：不是儲存更多資料，而是讓每一個概念都有來路、有關係、有後續。
