01｜核心概念

什麼是 AI Loop Engineering？

AI Loop Engineering 是一種設計 AI 智能體循環流程的工程方法，重點不是讓 AI 單次回答， 而是讓 AI 在可控流程中持續執行、觀察、評估、修正，直到完成任務或觸發停止條件。

🎯

從單次指令變成任務系統

Prompt Engineering 解決「怎麼問」，Loop Engineering 解決「如何讓 AI 持續完成任務」。

🔁

閉環：執行、觀察、修正

AI 每次行動後都讀取結果，根據測試、評分或工具回饋調整下一步。

🛑

必須可停止、可追蹤

好的 Loop 必須具備最大輪數、成本限制、狀態停滯偵測與人工接手機制。

一句話理解： Prompt 是「指令」，Loop 是「系統」；Prompt 讓 AI 開始工作，Loop 讓 AI 持續完成工作。
02｜觀念轉變

從 Prompt Engineering 到 Loop Engineering

傳統 AI 使用模式像是在帶實習生：人類必須不斷下指令、檢查、修正。 Loop Engineering 則把這些反覆動作設計成可以自動運轉的工程流程。

比較面向 Prompt Engineering Loop Engineering
核心問題 這句提示詞怎麼寫得更好？ 這套循環流程如何穩定執行？
任務型態 單次回答、單次生成、短任務 多步驟任務、工具調用、長任務、自動化工作流
AI 角色 回答者、內容生成者 執行者、觀察者、修正者、協作者
人類角色 反覆下指令與修正的人 系統設計者、規則制定者、品質把關者
品質來源 Prompt 品質與模型能力 模型能力 + 狀態管理 + 評估器 + 工具 + 停止條件
主要風險 回答不精準、格式不符 死循環、成本失控、工具誤用、上下文混亂
03｜核心原理

一個可落地的 AI Loop 需要哪些元件？

AI Loop 的關鍵不是「讓 AI 一直跑」，而是讓它在規則、工具、評估與停止條件之內有效運轉。

01

Goal

定義任務目標、完成標準、限制條件與輸出形式。

02

Plan

拆解任務、建立步驟、決定工具與執行順序。

03

Action

搜尋、讀寫檔案、執行測試、呼叫 API 或操作工具。

04

Observation

取得工具回傳、錯誤訊息、測試結果或使用者回饋。

05

Evaluation

檢查任務是否達標，分析品質、錯誤與下一步修正方向。

06

Stop / Continue

成功則停止；未完成則重試、換策略、交人工或終止。

實務原則： 每一個 Loop 都應該回答四個問題：現在做到哪裡？下一步做什麼？怎麼判斷做得好不好？什麼時候停止？
04｜流程圖

AI Loop 的標準執行流程

以下流程可以作為大多數 AI Agent 工作流的基礎架構，從軟體開發、文件生成到教材設計都能套用。

AI Loop Workflow
1
接收目標定義任務、範圍、完成標準與限制。
2
任務規劃拆解子任務，決定工具與順序。
3
建構上下文讀取文件、RAG、歷史摘要與狀態。
4
執行行動搜尋、修改、測試、呼叫 API。
5
觀察回饋讀取錯誤、測試結果與工具輸出。
6
品質評估比對標準，判斷是否達標。
7
策略修正若失敗，重新規劃或換工具。
8
停止或交人工成功、超限、無進展或高風險時停止。
05｜系統架構

AI 智能體的三層架構體系

Loop Engineering 不是孤立存在的，它通常建立在 Context Engineering 與 Agent Engineering 之上。

Layer 3 Loop Engineering

全局調度與循環控制層

決定由誰執行、何時執行、何時停止、何時切換 Agent、何時交給人類。 它像是整個 AI 工作流的總控台與調度器。

Layer 2 Agent Engineering

智能體執行與工具控制層

管理單一 Agent 的穩定運行，包括工具調用、錯誤重試、沙箱隔離、 權限控制、輸入輸出規則與例外處理。

Layer 1 Context Engineering

上下文、記憶與知識管理層

管理 AI 當下能看到的資訊，例如 RAG、文件片段、歷史摘要、 任務狀態、長短期記憶與外部資料來源。

簡單比喻： Context Engineering 像記憶體，Agent Engineering 像作業系統，Loop Engineering 像調度器。
06｜安全煞車

避免 AI 失控的三道煞車機制

Loop 系統最重要的不是「跑得久」，而是「知道何時該停」。

⏱

最大迭代輪數

設定絕對上限，例如最多 8 輪、20 輪或 100 輪。 到達上限就強制停止，避免死循環。

💰

Token 與成本限制

每輪記錄 Token、成本與執行時間；超過預算時停止或交人工判斷。

📉

狀態停滯偵測

若連續幾輪測試結果、錯誤訊息或品質分數沒有改善，就停止並回報原因。

if success_criteria_met: stop("success") elif iteration_count >= max_iterations: stop("max_iterations_reached") elif cost >= budget_limit: stop("budget_exceeded") elif no_progress_rounds >= 3: escalate_to_human("no_progress") elif high_risk_action_detected: request_human_approval() else: continue_loop()
07｜使用場景

哪些任務最適合導入 AI Loop Engineering？

最適合導入的任務通常具備三個特徵：重複性高、可驗證、失敗後可修正。

💻

軟體開發

讀 issue、修 Bug、跑測試、修錯、產生 PR 說明。

📄

技術文件

搜尋資料、建立大綱、生成初稿、補強範例、檢查格式。

🎓

教材開發

依課綱、年級與主題產生教案、學習單、Rubric 與互動教材。

☎️

客服知識庫

檢索回答、標記缺漏、收集人工修正、回饋知識庫。

🏢

企業流程

報表檢查、資料稽核、合約初審、風險攔截與人工覆核。

🔍

研究整理

查資料、比對來源、摘要重點、產出報告與參考資料。

08｜實作範例

Bug 修復 Loop：從 Issue 到測試通過

以下是最典型的 AI Loop Engineering 實作案例，適合用在 Codex、Claude Code、GitHub Copilot coding agent 或自建 Agent 系統。

任務流程

讀取 Issue

理解問題描述、重現條件與預期結果。

分析程式碼

尋找相關模組、函式、測試與錯誤來源。

修改與測試

進行最小修改，執行測試並收集錯誤。

反思與修正

根據錯誤 log 重新調整策略。

輸出結果

測試通過後產生變更摘要與 PR 說明。

Loop 設計規格

Goal: 修復 GitHub issue #128，並確保所有測試通過。 Tools: - read_file - edit_file - run_tests - inspect_logs Evaluation: - 單元測試全部通過 - 不破壞既有 API - 修改範圍合理 - 有清楚變更摘要 Stop: - 測試全部通過 - 超過 8 輪 - 連續 3 輪無進展 - 修改到高風險檔案 - 成本超過預算
09｜落地範本

可直接套用的 AI Loop Engineering 範本

將任何任務轉成 Loop 時，可以先填寫以下欄位，確認任務、工具、狀態、評估與停止條件都明確。

單一 Agent Loop 範本

[Task Name] 例如：自動生成 AI 技術文件 [Goal] 根據指定主題與來源，生成一份結構完整、可讀性高、可發布的 HTML 技術文件。 [Inputs] - 主題 - 來源資料或連結 - 讀者對象 - 格式需求 - 限制條件 [Tools] - Web search - File reader - Code editor - HTML generator - Link checker [State] - 已完成章節 - 已使用來源 - 已驗證內容 - 錯誤紀錄 - 輪數 / 成本 / 時間 [Evaluation] - 章節是否完整 - 是否有具體範例 - 是否符合讀者程度 - 是否有錯誤資訊 - HTML 是否可正常開啟 [Stop Conditions] - 文件完成且通過檢查 - 超過最大輪數 - 成本超標 - 連續多輪無進展 - 需要人工審核

多 Agent 協作範本

Planner: 拆解任務，規劃執行順序。 Researcher: 搜尋資料，整理來源與重點。 Builder: 生成內容、修改程式或建立文件。 Reviewer: 檢查品質、指出缺漏與風險。 Verifier: 執行測試、格式檢查、規則驗證。 Loop Controller: 管理狀態、成本、輪數與停止條件。 決定繼續、重試、切換 Agent、交人工或結束。
10｜範例 Prompt

把 Prompt 寫成 Loop 的控制規格

好的 Loop Prompt 不只是要求 AI 產出內容，而是明確描述目標、工具、規則、評估方式與停止條件。

範例 1：Bug 修復 Loop Developer Agent 
你是一個負責修復程式問題的 AI Agent。

任務目標：
修復 GitHub issue #128，並確保所有相關測試通過。

你可以執行的工具：
1. 讀取專案檔案
2. 修改專案檔案
3. 執行測試
4. 查看錯誤 log

工作規則：
- 先閱讀 issue 與相關程式碼，建立修復假設。
- 每一輪修改後都必須執行測試。
- 如果測試失敗，請分析錯誤原因，重新調整策略。
- 每輪都要更新目前狀態摘要。
- 如果連續 3 輪沒有進展，請停止並回報可能原因。
- 如果測試全部通過，請輸出：
  1. 修改摘要
  2. 變更檔案
  3. 測試結果
  4. 建議 PR 說明
範例 2：技術文件 Loop Documentation Agent 
你是一個技術文件生成智能體。

任務目標：
針對「AI Loop Engineering」撰寫一篇完整的 HTML 技術文件。

文件必須包含：
- 定義與核心概念
- 原理與三大要素
- 三層架構體系
- 流程圖與架構圖
- 常見使用場景
- 一個實作範例
- 落地導入流程
- 注意事項
- 導入檢查表
- 參考資料

工作規則：
- 先建立大綱。
- 再逐節生成內容。
- 每完成一節都檢查是否遺漏關鍵資訊。
- 完成初稿後，執行一次完整檢查。
- 最後輸出完整 HTML，不要只輸出摘要。
範例 3：教材開發 Loop Education Agent 
你是一個教育內容設計 Agent。

任務目標：
根據指定年級與課程主題，生成一套可用於教學的教材內容。

輸入條件：
- 年級：國小高年級
- 科目：自然與科技
- 主題：AI Loop Engineering 初探
- 對象：教師研習或資優課程

產出內容：
- 教學目標
- 核心概念解釋
- 流程圖
- 學習活動設計
- 課堂提問
- 學習單方向
- 評量方式

工作規則：
- 先列出教材架構。
- 每輪都檢查學生是否能理解。
- 若內容過難，請自動簡化語言並提供例子。
- 最後輸出成結構化 HTML。
範例 4：審查與評估 Loop Reviewer Agent 
你是一個 AI 工作流審查 Agent。

任務目標：
審查目前的 AI Loop 設計是否具備可落地性與安全性。

請檢查：
1. 任務目標是否明確
2. 是否有狀態管理
3. 是否有可用工具清單
4. 是否有評估標準
5. 是否有停止條件
6. 是否有成本限制
7. 是否有高風險操作的人類審核
8. 是否有 tracing / logging
9. 是否有 fallback 機制

輸出格式：
- 優點
- 缺漏
- 風險等級
- 修改建議
- 可落地版本的流程修正版
11｜注意事項

導入 AI Loop Engineering 前必須知道的風險

Loop Engineering 的價值來自自動化，但風險也來自自動化。以下是導入時最常見的錯誤。

⚠️

不要放任 AI 自動跑

Loop 不是放任 AI，而是將 AI 放進可控的規則與流程中。

📏

沒有評估器就不可靠

沒有明確評估標準，AI 只會一直「看起來有進展」。

🧩

多 Agent 不一定更好

Agent 越多，溝通成本、Token 成本與除錯難度也越高。

🧠

上下文要管理

Context 太少會失準，太多會混亂，必須摘要、篩選與更新。

🔐

工具權限要分級

讀取、寫入、刪除、寄信、金流與部署都需要不同安全等級。

📊

Tracing 不可省略

沒有 log 與 trace，就無法重現錯誤，也無法持續優化。

12｜落地實踐

如何一步一步落地 AI Loop Engineering？

建議先從低風險、高重複、可驗證的任務開始，再逐步加入工具、評估、監控與治理。

第一步：選擇低風險任務

例如技術文件整理、單元測試生成、Issue 分類、教材初稿、資料格式檢查。

第二步：拆成 Loop

把任務拆成搜尋、規劃、執行、觀察、評估、修正、停止等階段。

第三步：建立 MVP

先做 Goal → Agent → Tool → Evaluation → Stop 的最小可行版本。

第四步：加入狀態與紀錄

記錄每輪輸入、輸出、工具呼叫、錯誤、成本、評分與停止原因。

第五步：加入安全治理

加入最大輪數、預算、權限分級、人工審核、Guardrails 與 fallback。

第六步：建立評估資料集

用固定案例測試每次調整後的 Loop 是否真的變好。

第七步：產品化與複用

將成熟 Loop 做成可重複使用的工作流，例如 Bug Fix Loop、Documentation Loop、Lesson Plan Loop。

落地關鍵： 不要先追求全自動，而是先追求「小範圍可控、自動化可驗證、失敗可停止」。

AI Loop Engineering 專案檢查清單

在正式導入前，可用這份清單檢查 Loop 是否具備基本可用性與安全性。

  • 任務目標是否明確？
  • 是否定義完成標準？
  • 是否定義失敗條件？
  • 是否設定最大迭代次數？
  • 是否設定 Token / 成本預算？
  • 是否有評估機制？
  • 是否有狀態管理？
  • 是否有工具權限分級？
  • 高風險操作是否需要人工審核？
  • 是否有 tracing / logging？
  • 是否能偵測連續無進展？
  • 是否有 fallback 與人工接手機制？
  • 是否建立固定評估資料集？
  • 是否能重現每一輪決策？

總結：Prompt 是指令，Loop 是系統

AI Loop Engineering 的真正價值，不是讓 AI 一次就做對，而是把 AI 放進一套可觀察、 可評估、可修正、可停止、可治理的循環系統。對專案開發而言，它能將大量重複性的 查找、嘗試、修正、測試與整理工作交給 AI，在人類設定的規則內自動推進。

未來的核心能力不只是會寫 Prompt，而是能設計穩定的 AI Loop： 讓 AI 自動工作，但不失控；讓 AI 持續改進，但有邊界；讓人類退到更高價值的位置， 負責設計、判斷、審核與決策。
